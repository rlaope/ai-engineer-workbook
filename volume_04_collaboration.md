# Volume 4 — 협업과 의사소통

> 이 권이 끝나면 PM·연구자·SRE·데이터팀과의 인터페이스에서 *어떤 정보를 어떤 형식으로 주고받아야 하는지* 를 답할 수 있게 됩니다.

AI 엔지니어의 결과물은 혼자 만들어지지 않습니다. 제품 매니저는 *비즈니스 가치* 를, 연구자는 *모델 정확도* 를, SRE 는 *지연/가용성* 을, 데이터팀은 *데이터 품질* 을 우선시합니다. 같은 모델을 두고도 우선순위가 다르므로, 의사소통 실패는 잘못된 모델을 잘못된 환경에 배포하는 결과로 이어집니다. 이 권은 직무 간 인터페이스의 표준 산출물과 회의 패턴을 정리합니다.

## 이 권을 읽기 전에

기술 학습 자료의 대부분은 *코드와 모델* 에 집중합니다. 그러나 산업 현장에서 AI 엔지니어가 시간을 가장 많이 쓰는 영역 중 하나는 *다른 직무와의 협업* 입니다. 시간 통계로 25% 가 코드, 15-25% 가 회의·동기화, 10-15% 가 문서 작성에 투입된다는 사실은 Vol 1 의 *시간 비율* 챕터에서 이미 확인했습니다.

이 권은 그 *코드 외 시간* 을 효과적으로 만드는 도구상자입니다. 같은 8 시간을 *모호한 회의·답변 없는 슬랙 메시지·되돌아온 코드 리뷰* 에 쓰는 사람과 *명확한 모델 카드·짧고 결정적인 RFC·표준화된 인시던트 대응* 에 쓰는 사람의 차이가 *분기 단위 결과* 에서 매우 큰 격차를 만듭니다.

협업은 *기술 외 영역* 처럼 보이지만, 본질은 *시스템 사고의 확장* 입니다. 코드 컴포넌트 사이의 인터페이스를 명확히 정의하는 사고가 *인간 컴포넌트 사이의 인터페이스* 에도 그대로 적용됩니다. 이 권을 읽으면서 자신의 *팀 인터페이스* 를 그려 보시기 바랍니다.

## 1. 인접 직무의 우선순위 지도

### 1.1 다섯 직무의 우선순위

AI 엔지니어가 가장 자주 협업하는 다섯 직무와 그들의 우선순위:

| 직무 | 가장 신경 쓰는 것 | 자주 묻는 질문 |
|------|------------------|---------------|
| PM (제품 매니저) | 비즈니스 가치·사용자 만족도 | "이게 매출에 어떻게 기여하는가?" |
| AI 리서처 / ML 엔지니어 | 모델 정확도·일반화 | "최신 SOTA 와 비교해서 어떤가?" |
| SRE / DevOps | 지연·가용성·비용 | "P95 가 SLA 안에 들어오는가?" |
| 데이터 엔지니어 | 데이터 품질·신선도 | "이 데이터의 신뢰도는?" |
| 보안·컴플라이언스 | PII·규제 준수 | "이 모델이 규제를 위반하지 않는가?" |

같은 *AI 시스템 출시* 라는 사건도 다섯 직무가 보는 *측면* 이 모두 다릅니다. PM 은 *전환율*, 리서처는 *정확도 메트릭*, SRE 는 *오류율·지연*, 데이터 엔지니어는 *데이터 신선도*, 보안은 *PII 누출 가능성* 에 집중합니다.

좋은 AI 엔지니어는 *각 직무의 시각* 을 미리 인식하고, 회의·문서·코드 리뷰에서 *해당 시각을 미리 다룹니다*. 그러면 다섯 직무 모두에게 *우리의 우려가 반영되었다* 는 인상을 주어 합의가 빠르게 이뤄집니다.

### 1.2 우선순위 차이로 발생하는 갈등

흔한 갈등 패턴:

- *PM 은 "이번 주에 출시"* 를 원하는데 *리서처는 "정확도 95% 까지 학습 더"* 를 원함
- *SRE 는 "GPU 비용 줄이라"* 고 하는데 *AI 엔지니어는 "큰 모델이 정확도가 좋다"* 고 주장
- *데이터 엔지니어는 "데이터 변경에 1 주"* 가 필요한데 *AI 엔지니어는 "내일까지 새 데이터로 학습"* 을 요청

이런 갈등은 *어느 한 쪽이 틀린 것* 이 아니라 *다른 우선순위가 충돌한 것* 입니다. 해결의 핵심은 *공통 측정 기준* 을 만드는 일입니다.

```
정확도 1%p 향상 → 매출 X 원 증가
지연 100ms 단축 → 사용자 이탈 Y% 감소
GPU 비용 50% 절감 → 마진 Z%p 개선
```

이런 변환식이 합의되면, *우선순위 차이가 정량 비교* 가 가능해져 결정이 빨라집니다.

### 1.3 챕터 정리

다섯 인접 직무는 *각자 다른 우선순위* 를 가지며, 같은 사건도 다른 측면으로 봅니다. AI 엔지니어는 *각 직무의 시각을 미리 인식* 하고, 충돌 시 *공통 측정 기준* 으로 변환해 결정을 빠르게 만듭니다.

## 2. 모델 카드와 실험 보고서

### 2.1 모델 카드의 표준 양식

*모델 카드 (Model Card)* 는 모델을 다른 사람에게 전달할 때 함께 따라가는 *표준 메타데이터 문서* 입니다. 2018 년 Margaret Mitchell 등이 제안한 양식이 산업 표준이 되었으며, Hugging Face Hub 에 올라오는 모든 모델이 이 양식을 채택합니다.

```markdown
# Model Card: my-classifier-v3

## Overview
- 작업: 영화 리뷰 감정 분류 (긍정/부정)
- 베이스 모델: distilbert-base-uncased
- 학습 데이터: IMDB 25K + 사내 리뷰 5K
- 학습 일자: 2026-04-15
- 학습 책임자: @alice

## Performance
- 검증 정확도: 92.3%
- 사내 평가 셋: 88.5%
- P95 추론 지연: 35ms (T4 GPU)
- 모델 크기: 270MB

## Intended Use
- 일반 영화 리뷰 분류 (가능)
- 사내 제품 리뷰 분류 (가능 — 5K 미세조정 포함)
- 단문 리뷰 (10 단어 미만) — 정확도 낮음 (사용 비추천)
- 한국어 리뷰 — 학습 데이터에 없음 (사용 금지)

## Limitations
- 풍자·반어법에 약함
- 평점이 5/5 인 부정 리뷰 (ironic) 를 잘못 분류
- 영문 외 언어는 거의 동작하지 않음

## Training Details
- 학습률: 2e-5, AdamW, cosine decay
- 배치: 16, 3 epochs
- 학습 비용: 1 H100 × 2 시간 ≈ $4

## Safety & Ethics
- PII 학습 없음 (리뷰 텍스트만)
- 편향 검사: 영화 장르별 정확도 격차 ±3%p 이내
```

이 양식이 권장되는 이유는 *받는 사람이 항상 같은 위치에서 같은 정보를 찾을 수 있어서* 입니다. 새 모델이 도착했을 때 *Performance·Intended Use·Limitations* 만 보면 *이 모델을 쓸지 말지* 즉시 결정할 수 있습니다.

### 2.2 실험 보고서

실험을 끝내고 결과를 공유할 때의 표준 양식:

```markdown
# 실험: distilbert-base vs all-MiniLM-L6 (RAG 임베딩)

## 가설
all-MiniLM-L6-v2 가 distilbert-base 보다 우리 도메인에서 검색 정확도가 5%p 이상 높을 것

## 설정
- 평가 셋: 사내 FAQ 1000 쌍 (질의-답변)
- 메트릭: Top-3 Recall
- 임베딩 모델 외 모든 조건 동일

## 결과
| 모델 | Top-3 Recall | P95 임베딩 시간 | 메모리 |
|------|--------------|----------------|--------|
| distilbert-base | 78.2% | 12ms | 270MB |
| all-MiniLM-L6 | 84.5% | 4ms | 90MB |

## 결론
all-MiniLM-L6 채택. 정확도 +6.3%p, 속도 3 배, 메모리 1/3.

## 한계
- 사내 도메인 외 일반화는 검증 안 됨
- 멀티링귀얼 케이스는 별도 실험 필요

## 다음 단계
- 프로덕션 라우팅 변경 RFC 작성
- 1 주일 섀도우 트래픽 검증 후 전환
```

이 양식이 *동료가 5 분 안에 결과를 이해하고 의사결정할 수 있게* 만듭니다. 길이는 1-2 페이지가 표준.

### 2.3 챕터 정리

모델 카드는 *모델을 전달할 때 함께 따라가는 메타데이터*, 실험 보고서는 *실험 결과를 1-2 페이지로 압축한 문서* 입니다. 두 양식을 표준화하면 *팀의 정보 흐름 효율* 이 크게 올라갑니다.

## 3. OKR/KPI → 모델 메트릭 변환

### 3.1 비즈니스 메트릭과 모델 메트릭의 격차

PM 이 *전환율 5% 개선* 을 OKR 로 가져오면, AI 엔지니어는 이를 *어떤 모델 메트릭이 어느 만큼 개선되어야* 그 결과가 나오는지 변환해야 합니다.

이 변환은 종종 *직접 측정이 어려워* 추정이 필요합니다. 다음 패턴이 자주 사용됩니다.

**1. 오프라인-온라인 변환표.** 과거 실험에서 *오프라인 메트릭 변화 X 가 온라인 메트릭 변화 Y* 를 만들었던 사례를 표로 정리.

**2. A/B 테스트 결과.** 작은 변경의 A/B 결과로 *오프라인-온라인 비례 계수* 를 추정.

**3. 프록시 메트릭.** 직접 측정이 어려운 경우, *측정 가능한 프록시* (예: 클릭률·체류 시간) 로 대체.

### 3.2 메트릭 트리

복잡한 시스템에서는 *최상위 비즈니스 메트릭* 부터 *세부 모델 메트릭* 까지를 *트리* 로 구성합니다.

```
[최상위] 월 매출
   |
   +-- 전환율
   |     |
   |     +-- 추천 정확도 (오프라인 NDCG)
   |     +-- 추천 다양성 (Item Coverage)
   |     +-- 응답 속도 (P95 지연)
   |
   +-- 사용자 유지율
         |
         +-- 추천 만족도 (👍/👎)
         +-- 환각 발생률
```

이 트리가 있으면 *어떤 모델 메트릭이 어떤 비즈니스 메트릭에 기여* 하는지 한눈에 보이고, 우선순위 설정이 명확해집니다.

### 3.3 챕터 정리

OKR/KPI → 모델 메트릭 변환은 *오프라인-온라인 변환표·A/B 결과·프록시 메트릭* 의 세 도구로 만듭니다. 메트릭 트리로 *비즈니스 메트릭과 모델 메트릭의 연결* 을 시각화하면 우선순위가 명확해집니다.

## 4. 롤아웃 계획서

### 4.1 롤아웃의 단계

새 모델·새 시스템을 배포할 때는 *한 번에 전체 사용자에게 노출* 하면 위험합니다. 표준 롤아웃 단계:

1. **내부 검증** — 사내 직원·QA 만 사용
2. **카나리 (1-5%)** — 일부 사용자에게만 새 시스템 노출
3. **점진 확장 (10% → 50%)** — 메트릭 모니터링 하면서 확장
4. **전체 (100%)** — 옛 시스템 종료

각 단계마다 *진행/중단 기준* 을 미리 정의해 둬야 합니다.

### 4.2 롤아웃 계획서 양식

```markdown
# 롤아웃 계획: 새 RAG 모델 (v3)

## 단계와 일정
- D+0 (내부): 5 일
- D+5 (카나리 5%): 3 일
- D+8 (25%): 3 일
- D+11 (50%): 3 일
- D+14 (100%): 완료

## 모니터링 메트릭
- P95 응답 지연 < 800ms (이전 700ms)
- 환각율 < 3% (이전 4%)
- 사용자 만족도 (👍/👎) 비율 > 85% (이전 82%)

## 중단 기준 (Rollback 트리거)
- P95 지연 > 1500ms 5 분 이상 지속
- 환각율 > 6%
- 만족도 비율 < 75%
- 5xx 오류율 > 1%

## 롤백 절차
1. 트래픽 라우터를 옛 시스템으로 즉시 전환 (1 분)
2. Slack #incident 채널 알림 (자동)
3. 사후 분석 작성 (24 시간 안)

## 책임자
- 롤아웃 책임: @alice
- 온콜 SRE: @bob
- 의사결정 권한: @charlie (PM)
```

이 계획서가 미리 합의되면 *인시던트 발생 시 의사결정 시간* 이 크게 단축됩니다.

### 4.3 챕터 정리

롤아웃은 *내부 → 카나리 → 점진 → 전체* 의 4 단계로 진행하며, 각 단계마다 *진행/중단 기준* 을 미리 정의합니다. 롤아웃 계획서는 *책임자·메트릭·중단 기준·롤백 절차* 가 모두 들어간 표준 문서입니다.

## 5. 연구자와의 협업

### 5.1 시간 지평의 차이

AI 리서처의 한 프로젝트는 6 개월-2 년, AI 엔지니어의 한 프로젝트는 1-3 개월. 이 시간 지평 차이가 *마감 동기화* 를 어렵게 만듭니다.

협업 패턴:

- **분기 단위 동기화** — 분기 시작 시 리서처가 *이번 분기 후보 모델* 을 공유
- **모델 평가 게이트** — AI 엔지니어가 *프로덕션에 적합한지* 평가
- **점진 도입** — 합격한 모델만 카나리부터 시작

### 5.2 연구자의 결과를 프로덕션으로 옮기기

연구 코드와 프로덕션 코드 사이의 *간격* 이 큰 경우가 많습니다. 흔한 차이:

- 연구: *Jupyter 노트북, 단일 실행, GPU 한 장*
- 프로덕션: *모듈화된 Python, 동시 실행, 멀티 GPU/오토스케일*

AI 엔지니어가 연구 코드를 *프로덕션 가능한 형태로 리팩토링* 하는 일이 흔합니다. 이 과정에서 *결과가 미묘하게 달라지지 않도록* 다음을 점검:

- 시드 고정 (학습·평가 모두)
- 데이터 전처리 일치
- 정밀도 (FP32 → FP16) 변경 시 결과 비교
- 배치 크기 변화의 영향 측정

### 5.3 챕터 정리

리서처와 엔지니어의 *시간 지평·코드 스타일* 차이가 협업의 마찰점입니다. *분기 단위 동기화·모델 평가 게이트·점진 도입* 과 *연구→프로덕션 리팩토링 점검* 으로 마찰을 줄입니다.

---

## 6. SRE 와의 협업

### 6.1 인시던트 대응 흐름

AI 시스템 인시던트의 표준 대응 흐름:

```
[감지] 알람·사용자 신고
   ↓
[Triage] 영향 범위·심각도 분류 (5 분)
   ↓
[1차 대응] 롤백·트래픽 차단·우회 (5-30 분)
   ↓
[근본 원인 조사] 로그·메트릭·트레이스 분석 (1-N 시간)
   ↓
[수정] 패치·재배포 또는 다른 우회
   ↓
[사후 분석] Postmortem 문서 작성 (24-72 시간)
   ↓
[재발 방지] 모니터링·알람 강화·테스트 추가
```

각 단계의 *책임자와 채널* 을 미리 정해 두면 *인시던트 시 시간* 이 크게 단축됩니다.

### 6.2 런북 (Runbook)

자주 발생하는 인시던트는 *대응 절차* 를 *런북* 으로 미리 작성합니다.

```markdown
# Runbook: GPU OOM 발생 시

## 증상
- 서버 로그에 `CUDA out of memory` 반복
- 응답 5xx 비율 급증

## 1차 대응 (5 분 안)
1. 트래픽을 50% 줄임 (Load Balancer)
2. 배치 크기를 4 → 2 로 임시 축소
3. KV 캐시 최대 토큰 수 절반으로

## 진단 절차
1. `nvidia-smi` 로 메모리 사용량 확인
2. 현재 활성 요청 수와 평균 토큰 수 확인
3. 최근 모델·설정 변경 이력 확인

## 근본 원인 후보
- 새 모델 배포 후 메모리 증가
- 사용자 입력 길이 분포 변화
- KV 캐시 누수 (재시작 후 해결되면 의심)

## 사후 작업
- 메모리 사용량 알람 임계값 재검토
- 배치/캐시 크기 자동 조정 스크립트 검토
```

런북이 잘 갖춰진 팀은 *새 인력도 온콜에 빠르게 투입* 할 수 있습니다.

### 6.3 챕터 정리

SRE 와의 협업은 *표준 인시던트 대응 흐름·런북·사후 분석* 으로 만들어집니다. 인시던트 빈도 자체를 줄이는 것이 가장 좋지만, 발생 시 *대응 시간* 이 사용자 영향에 직접 비례합니다.

---

## 7. 데이터팀과의 협업

### 7.1 데이터 계약 (Data Contract)

데이터팀이 제공하는 데이터의 *스키마·신선도·품질 보장* 을 명시한 문서입니다.

```markdown
# Data Contract: user_events

## Schema
| 컬럼 | 타입 | 설명 | Nullable |
|------|------|-----|----------|
| event_id | UUID | 이벤트 식별자 | No |
| user_id | UUID | 사용자 식별자 | No |
| event_type | enum | view/click/purchase | No |
| timestamp | datetime UTC | 이벤트 발생 시각 | No |
| metadata | JSON | 부가 정보 | Yes |

## SLA
- 신선도: 5 분 이내 (이벤트 발생 → 테이블 반영)
- 가용성: 99.9% (월간)
- 중복: 0.01% 이하

## 변경 통보
- 스키마 변경 시 14 일 전 공지
- Breaking change 는 30 일 전 공지

## 책임자
- 데이터팀: @data-team
- 소비자: @ai-team, @analytics-team
```

데이터 계약이 없으면 *데이터 변경이 모델을 조용히 망가뜨리는* 사고가 발생합니다.

### 7.2 챕터 정리

데이터팀과의 협업은 *데이터 계약* 을 통해 *스키마·SLA·변경 통보* 를 명시합니다. 이 문서가 *데이터 변경으로 인한 모델 사고* 를 방지하는 가장 강력한 도구입니다.

---

## 8. 회의 패턴과 비동기 의사소통

### 8.1 회의 종류와 빈도

AI 팀의 표준 회의 종류:

- **데일리 스탠드업** (15-30 분, 매일) — 어제·오늘·블로커
- **주간 동기화** (1 시간, 주 1 회) — 이번 주 결과·다음 주 계획
- **분기 OKR 리뷰** (2 시간, 분기 1 회) — 분기 목표 점검
- **모델 리뷰** (1 시간, 격주) — 새 모델·실험 결과 공유
- **인시던트 사후 분석** (1 시간, 발생 시) — 무엇이 잘못되었나

회의가 너무 많으면 *깊은 작업 시간* 이 부족해집니다. 좋은 팀은 *회의 시간을 의도적으로 통제* 하고, 비동기 의사소통을 우선합니다.

### 8.2 비동기 우선

비동기 의사소통의 도구:

- **Slack/Discord** — 빠른 질문·결정
- **Notion/Confluence** — 긴 결정·기록
- **PR 코멘트** — 코드 리뷰·기술 토론
- **RFC 문서** — 큰 결정의 합의

회의 전에 *문서로 미리 공유* 하고, 회의에서는 *문서를 읽은 사람들의 토론* 만 하면 *회의 효율* 이 크게 올라갑니다.

### 8.3 챕터 정리

회의는 *목적별 종류와 빈도* 를 정해 두고, 비동기 의사소통을 우선합니다. 회의 전에 *문서를 공유* 해 회의 자체는 토론에 집중하는 패턴이 효율적입니다.

## 권 정리

이 권에서 우리는 다음을 배웠습니다.

- 인접 다섯 직무 (PM·리서처·SRE·데이터팀·보안) 는 *각자 다른 우선순위* 를 가집니다.
- 모델 카드와 실험 보고서는 *정보 전달의 표준 양식* 입니다.
- OKR/KPI → 모델 메트릭 변환은 *메트릭 트리* 로 가시화합니다.
- 롤아웃은 *4 단계와 중단 기준* 으로 안전하게 진행합니다.
- 연구자와의 협업은 *시간 지평 차이* 를 인식한 동기화로 마찰을 줄입니다.
- SRE 와의 협업은 *인시던트 대응 흐름과 런북* 으로 표준화합니다.
- 데이터팀과의 협업은 *데이터 계약* 으로 변경 사고를 방지합니다.
- 회의는 *목적별 통제 + 비동기 우선* 으로 효율을 만듭니다.

가장 기억에 남겨야 할 한 줄은 **"협업은 시스템 사고의 인간 컴포넌트 확장이며, 인터페이스를 명확히 정의한 팀이 분기 단위 결과의 격차를 만든다."** 입니다.
