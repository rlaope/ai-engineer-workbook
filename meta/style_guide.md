# 스타일 가이드

이 문서는 모든 권에 일관되게 적용되는 톤·금칙어·검증 규칙을 정의합니다.

---

## 1. 톤

### 어미

- **격식체 "~입니다"** 일관 사용
- "한다", "하자", "해라", "해 보자" 같은 평어·청유체 금지
- "~이다" 같은 단정형도 격식체 본문에서는 회피

### 인칭

- 일반론은 *3 인칭* — "독자는 ~", "AI 엔지니어는 ~"
- 직접 권유는 *2 인칭 (격식)* — "~ 시기 바랍니다", "~ 해 주십시오"
- *1 인칭* "우리는" 은 권 정리·챕터 정리 정도에만 제한적 사용

### 강조

- 핵심 용어 첫 도입: **굵은체** + 이탤릭 정의 (`**용어** — *정의*`)
- 본문 강조: *이탤릭* 만
- 굵은체 남발 금지

### 마케팅 표현 금지

- "🚀", "엄청난", "굉장히", "대단한", "혁신적인" 등 감정 형용사 금지
- 사실로만 진술

---

## 2. 모호 표현 금지

다음 표현은 *구체적 수치·조건·예시*로 교체.

| 금칙 | 권장 |
|------|------|
| 보통은 ... | 산업 평균 X% 의 경우 ... |
| 일반적으로 ... | 표준 패턴은 ... |
| 대체로 ... | (구체 조건 + 결과) |
| 다양한 ... | (3-5 가지 명시) |
| 여러 가지 ... | (열거) |
| 많이 ... | (수치) |
| 빠르게 ... | (지연 ms 또는 처리량) |
| 큰 ... | (사이즈 정량) |

---

## 3. 비유 규칙

- *백엔드·웹 한정 비유 금지* — 독자는 *AI 가 아닌 일반 개발자*
- 사용 가능한 비유: *함수 호출·캐시·로그·파이프라인·DB·API* 같은 일반 개발자 어휘
- 사용 금지: *Spring·Django·Rails* 같은 특정 프레임워크 비유

---

## 4. 사실 검증 — `[VERIFY]` 태그

다음 카테고리의 진술은 모두 검증 필요. 본문 작성 시 확신이 없으면 `[VERIFY: ...]` 태그를 남기고, 권 작성 후 일괄 검증.

- *날짜·연도* (예: "2017 년 발표된 ~ 논문")
- *수치* (예: "GPT-3 학습 비용 약 $4.6M")
- *논문 저자·제목·발표 학회*
- *모델 사이즈·파라미터 수* (예: "LLaMA 3 8B")
- *벤치마크 점수* (예: "MMLU 80%")
- *회사·인물 이름* (예: "Andrew Ng 의 Data-Centric AI")

검증 방법:
- 공식 페이지·논문 초록 직접 확인
- 위키·OpenReview·arXiv·Hugging Face 페이지

---

## 5. 코드 규칙

- 모든 코드는 *복사해 그대로 실행 가능*
- 첫 줄 주석으로 *권/챕터 표시 + 의존성 + 실행 방법*
- 코드 직후 *예상 출력* 또는 *결과 묘사* 한 단락
- PyTorch 2.4+ / Python 3.11+ 기준
- GPU 필요 시 명시 ("이 코드는 CUDA GPU 가 필요합니다")
- 외부 데이터 의존 시 *작은 더미 데이터* 함께 제공

---

## 6. 시각자료 규칙

- ASCII 다이어그램·표·수식만 사용
- *외부 이미지 링크 금지*
- *base64 PNG 금지*
- *mermaid 금지* (GitHub 렌더 호환성)
- 수식: GitHub 가 렌더링하는 `$...$` 또는 `$$...$$`
- 수식이 등장하면 *말로 한 줄 더 풀어 씀*

---

## 7. 금칙어

다음 단어가 본문에 등장하면 안 됨 (실수로 들어간 경우 즉시 교체):

- `과제`, `채용`, `chal`, `challenge`, `recruitment` — 챌린지·채용 종속 인상
- `Co-Authored-By: Claude` — 사용자 명시 금지
- `8 주`, `6 개월`, `12 개월`, `개월수` — 학습 일정 고정 표현 금지
- `🚀`, `🎯`, `💡` 등 이모지 — 사용자 명시 금지

검증: `grep -E "(과제|채용|chal|challenge|recruitment|Co-Authored-By|개월수|8 주|6 개월|12 개월)" volume_*.md`

---

## 8. 권 사이 일관성

- 같은 용어는 *모든 권에서 같은 정의*
- 새 용어 도입은 *그 용어가 처음 등장하는 권에서만*
- 다른 권 참조: `Vol N (권 제목)`
- 권 간 의존성은 *선수 지식* 섹션에 명시

---

## 9. 분량 가이드

| 권 종류 | 챕터 수 | 권 분량 |
|---------|---------|---------|
| 입문 (직무·큰그림) | 8-10 | 1,500-2,000 |
| 수학 (이론 + NumPy) | 8-10 | 1,800-2,500 |
| ML/DL 핵심 | 8-12 | 2,000-2,800 |
| 응용 (CV·NLP·LLM) | 8-12 | 2,000-2,800 |
| 시스템 (서빙·인프라) | 8-12 | 2,200-3,000 |
| 운영·거버넌스 | 6-10 | 1,800-2,500 |

분량 자체보다 *밀도*가 중요. 부풀리기 금지.

---

## 10. 커밋 규칙

- 권 단위가 아닌 *3-5 권 batch commit*
- 메시지: `docs(volume_N1-N2): write/expand bodies for X, Y, Z`
- *Co-Authored-By: Claude* 라인 절대 금지
- 푸시 빈도: batch commit 마다

---

## 11. Edit vs Write 선택

- **Write** — 신규 권 (스켈레톤만 있는 권) 의 본문 첫 작성
- **Edit** — 기존 본문 보강 (새 섹션 삽입·기존 섹션 확장)
- 같은 파일에 여러 차례 손댈 때는 *Edit 사용을 우선*해 출력 토큰 절감
