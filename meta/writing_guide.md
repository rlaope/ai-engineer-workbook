# 본문 작성 가이드 (Writing Guide)

이 문서는 104 권의 *본문* 을 채울 때 따라야 할 규약입니다. 모든 권은 같은 구조·톤·품질 기준을 가져야 합니다.

---

## 1. 톤·말투

- **격식체 "~입니다" 일관 적용.** "한다", "하자" 같은 평어·청유체 금지.
- **모호한 표현 금지.** "보통은", "일반적으로", "대체로", "다양한", "여러 가지" 같은 표현은 구체 수치·조건·예시로 교체.
- **백엔드·웹 한정 비유 금지.** 일반 *AI 가 아닌 개발자* 가 독자입니다. 비유는 *함수 호출·캐시·로그* 같은 일반적 개발자 어휘만 사용.
- **이모지·마케팅 표현 금지.** "🚀", "굉장히", "대단한" 등 사용하지 않습니다.
- **선언이 아니라 설명.** "X 가 좋습니다" 보다 "X 는 Y 한 이유로 Z 한 상황에서 적합합니다" 같은 *근거*를 동반.

---

## 2. 권 본문 표준 구조

각 권은 *스켈레톤 그대로* 시작해 다음 순서로 확장합니다.

```
1. (기존) 권 표지 — 제목 + 한 줄 목적
2. (기존) ## 목적 — 3-5 문장
3. (기존) ## 선수 지식
4. (기존) ## 학습 결과 — 5-8 항목
5. (NEW) 권 도입 — "이 권을 읽기 전에" 한 단락 (300-600 자)
6. (NEW) ## 1. 첫 번째 챕터 제목
   - 도입 단락 — 왜 이 챕터를 읽는가 (200-400 자)
   - 개념 본문 — 정통 개념서 톤 (1500-3000 자)
   - 시각 자료 — ASCII 다이어그램 또는 표 (필요 시)
   - 코드 블록 — 실행 가능, 50-200 줄 (해당 챕터에 적합한 경우)
   - 챕터 정리 — 핵심 한 단락 (200 자 내)
7. ## 2. 두 번째 챕터 — (위와 같은 구조)
... (8-14 챕터)
8. (NEW) ## 권 정리 — 한 페이지 요약
9. (기존) ## 자가점검 키워드
10. (NEW) ## 자가점검 질문 — 5-7 문항 (스켈레톤의 키워드를 풀어쓴 질문)
11. (기존) ## 다음 권
```

### 분량 목표

- **권 전체**: 2,500–4,500 라인 (한국어 기준)
- **챕터당**: 200–400 라인
- **코드 블록**: 권당 최소 1개, 가능하면 챕터당 1개

## 3. 챕터 본문 작성 규약

### 3.1 도입 단락

다음 세 가지 중 하나로 시작합니다.

1. **문제 제기** — "이 챕터에서는 ~ 라는 질문에 답합니다."
2. **이전 챕터와의 연결** — "지난 챕터에서 ~ 를 보았습니다. 이번에는 그 ~ 가 ~ 로 어떻게 이어지는지를 다룹니다."
3. **역사적·실무적 맥락** — "2017 년 발표된 ~ 논문은 ~ 라는 발상을 처음 제시했습니다."

### 3.2 개념 본문

- **첫 정의는 명시적으로.** 새 용어는 굵게 + 정확한 정의 한 문장. *추측하게 두지 않습니다.*
- **수식은 LaTeX.** GitHub 가 렌더링하는 `$...$` 또는 `$$...$$` 사용.
- **수식이 등장하면 *말로* 한 줄 더 풀어 씁니다.** 예: `$\nabla L$` 는 *손실의 그래디언트, 즉 손실이 가장 빠르게 증가하는 방향* 입니다.
- **단계적 설명.** *왜 이게 필요한가 → 어떻게 작동하는가 → 어떤 한계가 있는가* 의 흐름.
- **반례·실패 사례 포함.** "이 방법이 무너지는 경우" 단락이 있을 때 이해가 깊어집니다.

### 3.3 시각 자료

ASCII 다이어그램·표·수식은 *말로 설명한 내용을 한눈에 다시 보여 주는* 역할입니다.

- ASCII 다이어그램: 큰 그림용
- 표: 비교가 본질일 때만
- 수식: 정확함이 본질일 때만

**금지:** 외부 이미지 링크, base64 PNG, mermaid 다이어그램(렌더 호환성). ASCII 만 사용.

### 3.4 코드 블록

```python
# Vol N — 챕터 제목
# 실행: python this_file.py  (또는 Jupyter)
# 의존성: torch>=2.4, numpy

import torch
import torch.nn as nn

# ... 본문 ...
```

- 모든 코드는 **그대로 복사해서 실행 가능**해야 합니다.
- 첫 줄에 권/챕터 표시 + 실행 방법 + 의존성 주석.
- 실행 결과(또는 그래프 모양 묘사) 를 코드 블록 *직후* 텍스트로 적어 줍니다.
- PyTorch 2.4+ / Python 3.11+ 기준.
- GPU 가 필요한 코드는 명시 ("이 코드는 CUDA GPU 가 필요합니다").

### 3.5 챕터 정리

각 챕터 마지막에 *한 단락 (200 자 내)* 으로 핵심을 압축합니다. *무엇을 배웠는가 + 다음 챕터가 어떻게 이어지는가*.

## 4. 권 정리 (Volume Summary)

권 끝에 *한 페이지 분량* 의 종합 정리를 둡니다.

```markdown
## 권 정리

이 권에서 우리는 다음을 배웠습니다.

- {핵심 1}
- {핵심 2}
...

가장 기억에 남겨야 할 한 줄은 **"{이 권의 한 줄 요약}"** 입니다.

다음 권은 {다음 권 제목} 입니다. 거기서는 {다음 권에서 다룰 핵심} 을 다룹니다.
```

## 5. 작성 순서

### 추천 순서: 학습 순서 그대로 (Volume 1 → 104)

이유:
- 의존성 그래프와 일치
- 독자가 *처음부터 끝까지* 읽을 가능성이 가장 높은 경로
- 본문 작성 중 일관성을 점검하기 쉬움

### 대안 1 — 임팩트 우선

다음 6 권을 먼저 완성해 *체험 가능한 상태* 를 빠르게 만든 뒤 나머지를 채우는 방식입니다.

`Volume 1, 32, 40, 41, 48, 49`

(직무 → Transformer → RAG → Agent → 추론 최적화 → 서빙)

### 대안 2 — 트랙 단위

한 트랙을 끝까지 채운 뒤 다음 트랙으로. 트랙 내 일관성이 가장 좋습니다.

## 6. 검증 체크리스트 (권 단위)

본문을 마친 권은 다음을 통과해야 합니다.

- [ ] 격식체 "~입니다" 일관 사용 (`grep -E "(한다|하자)\." | wc -l == 0`)
- [ ] 모호 표현 0건 (`grep -E "(보통은|일반적으로|대체로)"`)
- [ ] 금칙어 0건 (`grep -E "(과제|채용|chal|challenge|recruitment|Co-Authored-By|개월|6 개월|12 개월|8 주)"`)
- [ ] 모든 코드 블록 실행 검증 (Python 인터프리터로 직접 돌려 봄)
- [ ] 외부 이미지 링크 0건
- [ ] mermaid 0건
- [ ] 권 본문 분량 2,500 라인 이상
- [ ] 챕터 수가 스켈레톤의 *챕터 목차* 와 일치
- [ ] 자가점검 키워드와 본문에서 다룬 용어 일치

---

## 7. 커밋 규약

- 권 단위로 커밋: `docs(volume_NN): write body for {권 제목}`
- 메시지 본문에 *추가된 챕터 수·코드 블록 수·라인 수* 기록
- 한 커밋에 한 권 (대형 권이면 챕터 단위로 분할 가능)
- **`Co-Authored-By: Claude` 라인 절대 금지**

## 9. 참고

- 시각자료는 본문 보조이며, *문장 본문이 압도적으로 많아야* 합니다 (사용자 명시 요구).
- 표·코드보다 본문 텍스트가 풍부할 것.
- 한 권을 끝낸 뒤에는 *처음 정의한 용어를 본문 어디서든 다시 같은 의미로 쓰고 있는지* 검토.
