## 1. AI·ML·DL — 포함 관계와 정의

### 1.1 가장 흔한 혼동

뉴스·블로그·채용 공고에서 *AI*·*머신러닝*·*딥러닝* 이라는 세 단어는 거의 동의어처럼 쓰입니다. *AI 엔지니어 채용*·*머신러닝 모델 도입*·*딥러닝 기반 추천* 같은 표현이 같은 글 안에 섞여 등장합니다. 처음 분야에 들어오는 사람에게 이 혼용은 *어디서부터 배워야 하는가* 라는 질문을 어렵게 만듭니다.

이 혼동은 단지 일반인의 문제가 아닙니다. 회사 임원진이 *우리도 AI 를 도입해야 한다* 고 말할 때, 그 사람의 머릿속에 있는 *AI* 가 무엇인지를 정확히 알기 어렵습니다. 어떤 사람은 *데이터로 분류 모델 학습* 을 의미하고, 다른 사람은 *ChatGPT 같은 챗봇 도입* 을 의미하며, 또 다른 사람은 *RPA (Robotic Process Automation) 같은 단순 자동화* 를 의미할 수 있습니다. 같은 단어가 같은 회의 안에서 다른 의미로 사용되면, 프로젝트 정의 단계에서부터 큰 혼선이 생깁니다.

따라서 이 권의 첫 챕터는 *AI·ML·DL 을 정확히 구분* 하는 데 집중합니다. 이 구분이 명확해지면, 회의에서 *그 AI 는 정확히 어떤 의미인가요* 를 물을 수 있게 되고, 프로젝트 정의가 *훨씬 빠르고 정확* 해집니다.

이 세 단어는 *동의어가 아니라 포함 관계* 입니다.

```
+---------------------------------------------+
|  AI (Artificial Intelligence)               |
|  +---------------------------------------+  |
|  |  ML (Machine Learning)                |  |
|  |  +---------------------------------+  |  |
|  |  |  DL (Deep Learning)             |  |  |
|  |  |                                 |  |  |
|  |  |  - CNN, RNN, Transformer        |  |  |
|  |  |  - GAN, VAE, Diffusion          |  |  |
|  |  |  - LLM, Vision Transformer      |  |  |
|  |  +---------------------------------+  |  |
|  |                                       |  |
|  |  - 선형 회귀, 의사결정나무            |  |
|  |  - 랜덤 포레스트, XGBoost              |  |
|  |  - SVM, K-Means                       |  |
|  +---------------------------------------+  |
|                                             |
|  - 규칙 기반 시스템 (Expert System)         |
|  - 탐색 알고리즘 (A*, Minimax)              |
|  - 게임 트리, 제약 만족                     |
+---------------------------------------------+
```

이 그림은 세 가지 사실을 보여 줍니다.

첫째, *AI 가 가장 큰 원* 입니다. AI 는 머신러닝뿐 아니라 규칙 기반 시스템·탐색 알고리즘·제약 만족 문제 같은 *데이터로 학습하지 않는* 분야도 포함합니다. 1956 년 다트머스 회의에서 *Artificial Intelligence* 라는 단어가 처음 공식화되었을 때, 연구자들이 염두에 둔 것은 *기호적 추론·탐색·논리* 였지 데이터 기반 학습이 아니었습니다.

둘째, *머신러닝이 그 안에 들어가는 한 부분집합* 입니다. 머신러닝은 *데이터로부터 패턴을 학습하는* 모든 알고리즘을 가리키며, 선형 회귀·로지스틱 회귀·의사결정나무·랜덤 포레스트·SVM·K-Means 같은 *고전 ML* 알고리즘과 딥러닝을 모두 포함합니다.

셋째, *딥러닝은 머신러닝 안의 더 작은 부분집합* 입니다. 딥러닝은 *깊은 신경망 (여러 층의 신경망)* 을 사용하는 머신러닝이며, CNN·RNN·Transformer 같은 모든 현대 신경망 아키텍처가 여기에 들어갑니다. LLM·Vision Transformer·디퓨전 모델은 모두 *딥러닝의 응용 사례* 입니다.

### 1.2 정의를 한 문장씩

각 단어를 한 문장으로 정의합니다.

**AI (Artificial Intelligence)** — *인간의 지능적 행동을 컴퓨터로 흉내 내는 모든 기술의 총칭* 입니다. 데이터 학습 여부와 무관하며, 1956 년 다트머스 회의에서 개념적으로 출발했습니다.

다트머스 회의는 *AI 라는 분야의 공식 출발점* 으로 간주되는 사건입니다. John McCarthy·Marvin Minsky·Claude Shannon·Nathaniel Rochester 가 주최한 이 워크숍에서 *기계가 지능적 행동을 흉내 내게 만드는 일* 을 한 학문 분야로 정립하자는 합의가 이루어졌습니다. 그 결과로 *AI* 라는 용어가 만들어졌고, 이후 70 여 년 동안 분야의 이름으로 살아남았습니다.

**ML (Machine Learning)** — *데이터로부터 패턴을 학습해, 새로운 데이터에 대한 예측이나 결정을 만드는 알고리즘과 그 응용* 을 가리킵니다. 1950-60 년대부터 통계학과의 결합으로 발전했고, 1980-2010 년대에 본격적으로 산업에 적용되었습니다.

머신러닝이라는 단어 자체는 1959 년 Arthur Samuel 이 IBM 에서 *체커스 (서양 장기) 를 학습하는 프로그램* 을 만들면서 처음 사용했습니다. 그의 정의는 *명시적으로 프로그래밍되지 않고도 학습할 수 있는 능력을 컴퓨터에게 부여하는 연구 분야* 였습니다. 이 정의는 70 여 년이 지난 지금도 거의 그대로 통용됩니다.

**DL (Deep Learning)** — *여러 층의 인공 신경망을 사용하는 머신러닝* 입니다. 이론적으로는 1980 년대 역전파 알고리즘이 등장하면서 가능해졌지만, 2012 년 AlexNet 이후 GPU 컴퓨팅과 결합해 폭발적으로 발전했습니다.

딥러닝이라는 용어는 2006 년 Geoffrey Hinton 의 *Deep Belief Networks* 논문에서 본격적으로 사용되기 시작했습니다. 그러나 이 분야가 실제로 산업의 주류로 올라온 것은 2012 년 AlexNet 의 ImageNet 우승부터입니다. 그 사이의 약 30 년은 *AI 겨울* 이라 불리는 침체기였으며, 신경망 연구는 학계의 비주류로 밀려나 있었습니다.

이 세 정의는 *시간순으로 좁아지는* 흐름을 가집니다. AI 가 가장 오래된 큰 우산이고, ML 이 그 안에서 통계적 학습 분야로 분화되었고, DL 이 그 안에서 깊은 신경망 분야로 다시 분화되었습니다. 그러나 *현재의 무게중심* 은 정반대로, *가장 안쪽에 있는 DL* 에 있습니다. 이 역설을 이해하는 것이 다음 챕터의 주제입니다.

### 1.3 AI ⊃ ML ⊃ DL — 그러나 현재의 무게중심

포함 관계로만 보면 *AI 가 가장 크다* 고 생각하기 쉽지만, *현재 산업에서 가장 활발히 발전하고 가장 많은 자본·인재가 투입되는 영역* 은 가장 안쪽의 DL 입니다.

이는 다음 이유 때문입니다.

첫째, *대부분의 어려운 실세계 문제(자연어 이해·이미지 인식·음성 처리·코드 생성)* 에서 딥러닝이 다른 모든 접근을 압도했습니다. 이 영역에서는 규칙 기반·고전 ML 이 거의 자리를 잃었습니다. 2012 년 AlexNet 의 ImageNet 우승이 이미지 인식 분야의 결정적 전환점이었고, 2017 년 Transformer 논문이 자연어 처리 분야의 결정적 전환점이었으며, 2022 년 ChatGPT 가 *일반 대중에게 AI 를 알린* 결정적 전환점이었습니다. 이 세 사건 모두가 딥러닝의 응용입니다.

둘째, *컴퓨팅 자원의 발전*(GPU·TPU) 과 *데이터의 증가* 가 결합하면서, 깊은 신경망의 잠재력이 풀려났습니다. 1980 년대에 발견되었지만 묶여 있던 알고리즘들이 2010 년대 이후 진가를 발휘했습니다. *역전파 알고리즘* 은 1986 년 Rumelhart·Hinton·Williams 의 논문으로 정립되었지만, 당시의 컴퓨팅 자원으로는 깊은 신경망 학습이 사실상 불가능했습니다. 30 년 후 *NVIDIA GPU 의 보급 + 인터넷 데이터의 폭발* 이 결합되면서야 그 알고리즘이 본격적으로 활용되기 시작했습니다.

셋째, *파운데이션 모델 시대* (2020 년대) 에 들어, 거대한 사전학습 신경망을 다양한 작업에 적용하는 패러다임이 자리잡았습니다. 이 패러다임은 본질적으로 딥러닝의 응용입니다. 2018 년 BERT, 2020 년 GPT-3, 2022 년 ChatGPT, 2023 년 GPT-4, 2024 년 Claude 3 시리즈로 이어지는 흐름은 *모두 같은 딥러닝 아키텍처 (Transformer) 의 점진적 스케일업* 입니다.

따라서 현재 *AI 엔지니어* 라는 직무가 다루는 모델의 압도적 다수는 딥러닝 모델입니다. 그러나 *고전 ML 이 사라졌다* 는 의미는 아닙니다. 정형 데이터(테이블 형태) 에서는 XGBoost·LightGBM 같은 부스팅 알고리즘이 여전히 가장 강력하며, *딥러닝이 잘 안 되는 영역에서 고전 ML 이 살아남는다* 는 사실은 이 책의 Track C 전체에서 다룹니다. *Kaggle* 같은 데이터 과학 경진대회에서도 정형 데이터 문제는 여전히 부스팅 계열이 우승하는 경우가 많습니다.

또한 *규칙 기반 AI* 도 완전히 사라지지 않았습니다. 항공 관제·의료 진단 보조·세무·법률 같은 *규제가 엄격하고 결정 근거가 투명해야 하는* 영역에서는 규칙 기반 시스템이 여전히 사용됩니다. 최근에는 *LLM + 규칙 기반 시스템* 의 하이브리드 형태도 등장하고 있습니다 — 예를 들어 LLM 이 자연어 입력을 이해하면, 규칙 기반 시스템이 결정의 정합성·규제 준수를 검증하는 식의 결합입니다.

### 1.4 "AI" 라는 단어의 두 가지 의미

마지막으로, 일상 대화에서 *AI* 라는 단어가 의미하는 바는 두 가지로 갈라져 사용됩니다.

**좁은 의미의 AI (Narrow AI)** — 현재 우리가 만드는 모든 AI 시스템이 여기에 해당합니다. 특정 작업(번역·이미지 분류·문서 요약) 을 잘 수행하지만, 그 작업을 벗어나면 의미 있는 행동을 못 합니다. ChatGPT, Claude, Gemini 같은 LLM 도 *대화·텍스트 생성* 이라는 광범위한 작업을 잘하지만, 본질적으로는 좁은 의미의 AI 에 속합니다.

좁은 의미의 AI 는 *Narrow AI* 또는 *Weak AI* 라고도 불리며, 학계에서는 *ANI (Artificial Narrow Intelligence)* 라는 약자가 사용되기도 합니다. 핵심은 *학습한 작업의 범위 안에서만 의미 있는 행동* 을 한다는 점입니다. 바둑 챔피언을 이긴 *AlphaGo* 도 바둑 외에는 거의 아무 일도 못 하며, 이미지를 잘 만드는 *Stable Diffusion* 도 텍스트 요약은 못 합니다.

**일반 인공지능 (AGI: Artificial General Intelligence)** — 인간이 할 수 있는 거의 모든 지적 작업을 동등하거나 더 잘하는 가상의 시스템입니다. 아직 존재하지 않으며, 언제 가능한지에 대해서는 연구자들 사이에서도 큰 이견이 있습니다.

AGI 라는 용어 자체는 2005 년 전후로 학계에서 사용되기 시작했고, 이후 OpenAI·DeepMind·Anthropic 같은 연구소들이 명시적으로 AGI 를 *목표* 로 선언하면서 일반 대중에게도 알려졌습니다. AGI 가 언제 가능한지에 대한 추정은 *5 년 안* 부터 *영원히 불가능* 까지 매우 넓은 범위로 갈립니다. 이 추정의 차이는 *AGI 를 어떻게 정의하는가* 에 따라 크게 달라지므로, 누군가 *AGI 가 곧 온다* 고 말할 때는 *그가 의미하는 AGI 가 무엇인지* 를 먼저 확인해야 합니다.

뉴스에서 *AI 가 인간을 대체* 같은 표현이 등장할 때, 그 글이 *좁은 의미의 AI* 에 대해 말하는지 *AGI* 를 말하는지를 구분해 읽으면 과도한 기대나 공포에서 자유로워집니다. AI 엔지니어로서는 *현재 우리가 만드는 모든 AI 가 좁은 의미의 AI* 라는 사실을 명확히 인식한 채 일하는 것이 건강한 출발점입니다.

### 1.5 자주 마주치는 용어 정리

분야에 들어오면 *AI*·*ML*·*DL* 외에도 수많은 용어가 등장합니다. 이 챕터의 마무리로, 자주 마주치는 용어들을 정리합니다.

- **GenAI (Generative AI)** — *생성형 AI*. 이미지·텍스트·음성·비디오를 *새로 만들어 내는* AI. LLM·디퓨전 모델·GAN 등이 포함. 최근 2-3 년 사이 가장 많이 들리는 용어.
- **LLM (Large Language Model)** — *대규모 언어 모델*. GPT·Claude·Gemini·LLaMA 같은 거대 사전학습 텍스트 모델.
- **Foundation Model** — *파운데이션 모델*. LLM 을 포함한 더 넓은 범주. 비전·음성·다중모달 거대 모델 모두 포함.
- **Agent / AI Agent** — *에이전트*. LLM 이 도구를 사용하고 여러 단계 작업을 수행하는 시스템.
- **RAG (Retrieval-Augmented Generation)** — *검색 증강 생성*. LLM 이 답하기 전에 외부 문서를 검색해 컨텍스트로 주입하는 패턴.
- **MLOps** — *머신러닝 운영*. ML 시스템의 개발·배포·운영 자동화 분야.
- **AGI** — *일반 인공지능*. 위에서 다룸.
- **ASI (Artificial Superintelligence)** — *초지능*. 인간을 모든 영역에서 압도하는 가상의 시스템. 현재로선 사변적 개념.

이 용어들을 모두 *처음부터 깊이 이해할 필요는 없습니다*. 이후 권에서 각각이 본격적으로 등장하므로, 지금은 *이런 용어가 있다* 정도만 알아 두시면 충분합니다.

### 1.6 챕터 정리

AI·ML·DL 은 *동의어가 아니라 포함 관계* 입니다. AI 가 가장 넓은 우산이고, ML 이 데이터로부터 학습하는 부분집합이며, DL 이 깊은 신경망을 사용하는 더 좁은 부분집합입니다. 시간순으로는 AI → ML → DL 의 순서로 좁아지는 분화가 일어났지만, *현재의 무게중심은 정반대로* 가장 안쪽의 DL 에 있습니다. 정형 데이터 영역에서는 고전 ML 이 여전히 강력하며, 규제 영역에서는 규칙 기반 AI 도 살아남아 있습니다. 좁은 의미의 AI 와 AGI 의 구분은 뉴스를 정확히 읽기 위한 기본 도구입니다. 다음 챕터에서는 이 분야가 어떤 패러다임의 변천을 거쳐 지금 모습이 되었는지를 봅니다.

## 2. 패러다임의 변천 — 규칙 기반 → 통계 기반 → 표현 학습 → 파운데이션 모델

### 2.1 규칙 기반 시대 (1956 - 1980 년대 후반)

AI 의 첫 30 년은 *규칙 기반* 패러다임이 지배했습니다. 인간 전문가가 *지식과 규칙* 을 명시적으로 컴퓨터에 입력하면, 컴퓨터가 그 규칙을 *논리적으로 적용* 해 결정을 내린다는 발상입니다.

대표 사례는 *전문가 시스템(Expert System)* 입니다. 1970-80 년대 의료 진단·금융 의사결정·산업 자동화에 광범위하게 적용되었습니다. 예를 들어, 의료 진단 전문가 시스템은 *증상 → 질병* 의 매핑 수만 개를 의사들이 직접 입력하고, 환자의 증상 입력에 대해 그 규칙들을 조합해 질병을 추정했습니다.

가장 유명한 전문가 시스템 중 하나는 1976 년 Stanford 대학에서 개발된 *MYCIN* 입니다. 이 시스템은 약 600 개의 규칙으로 *세균 감염 진단과 항생제 처방* 을 지원했고, 일부 평가에서 *사람 의사보다 더 정확한 처방* 을 제시했습니다. 그러나 임상에 직접 도입되지는 못했는데, 이는 *법적 책임 문제·의사들의 거부감·기존 의료 시스템과의 통합 비용* 같은 *기술 외* 이유 때문이었습니다. 이 패턴은 70 년 후의 AI 도입에서도 반복적으로 관찰되는 *기술 외 도전* 의 첫 사례라 할 수 있습니다.

이 패러다임의 강점은 *결정 과정이 투명* 하다는 점입니다. 어떤 규칙이 어떤 결정을 만들었는지 추적할 수 있으므로, 규제가 엄격한 분야에서 유리했습니다. 1980 년대에는 미국·일본·유럽에서 전문가 시스템에 대한 막대한 투자가 일어났고, *5 세대 컴퓨터 프로젝트* (일본, 1982-1992) 같은 국가 단위 사업도 이 패러다임 위에서 진행되었습니다.

약점은 *지식 추출의 한계* 입니다. 인간 전문가의 *암묵적 지식* 을 명시적 규칙으로 옮기는 일은 극도로 어렵고, 도메인이 조금만 바뀌어도 규칙을 처음부터 다시 만들어야 했습니다. 이를 *지식 획득 병목 (knowledge acquisition bottleneck)* 이라 부릅니다. 이 병목 때문에 1990 년대 들어 전문가 시스템 시장이 급격히 위축되었습니다.

또 다른 약점은 *모호함 처리의 한계* 입니다. 의사가 진단을 내릴 때는 *증상 A 가 있으면 70% 확률로 질병 X* 같은 *확률적 사고* 를 하지만, 규칙 기반 시스템은 이런 확률적 추론에 약했습니다. 일부 시스템은 *certainty factor* 같은 우회 기법을 도입했지만, 본질적으로 *결정론적 규칙의 조합* 으로 *확률적 현실* 을 다루는 데 한계가 있었습니다.

이 시기의 *AI 겨울 (AI Winter)* 은 1987-1993 년경에 시작되어 약 10 년간 지속되었습니다. 전문가 시스템에 대한 기대가 무너지면서 *Lisp 머신* 시장도 붕괴되고, AI 연구에 대한 정부 자금이 대폭 삭감되었습니다. 이 시기에 학계는 통계적 방법론으로 점차 옮겨가기 시작했습니다.

### 2.2 통계 기반 시대 (1980 년대 후반 - 2010 년대 초)

규칙 기반의 한계가 드러나면서, *데이터로부터 패턴을 자동으로 학습* 하는 통계 기반 접근이 부상했습니다. 이 시기에 머신러닝이 본격적인 산업 분야로 자리잡았습니다.

대표 알고리즘은 다음과 같습니다.

- 의사결정나무 (Decision Tree) — ID3 (1986), C4.5 (1993), CART (1984)
- 랜덤 포레스트 (Random Forest, 2001)
- 서포트 벡터 머신 (SVM, 1995)
- 부스팅 (AdaBoost, 1995; GBM, 2001)
- 나이브 베이즈 (Naive Bayes)
- k-최근접 이웃 (k-NN)
- 잠재 디리클레 할당 (LDA, 2003)

이 알고리즘들은 *수치형·범주형 입력 특성* 을 받아 *예측* 을 내는 일에 매우 강력했습니다. 신용 평가·스팸 필터·추천 시스템·의료 영상 분석 같은 산업이 통계 기반 머신러닝 위에서 만들어졌습니다.

이 시대를 대표하는 산업 적용 사례는 *Netflix Prize* (2006-2009) 입니다. Netflix 가 *영화 추천 알고리즘의 정확도를 10% 개선* 하면 100 만 달러를 주겠다고 공표한 이 대회는, *행렬 분해* 기반 추천 알고리즘의 발전에 결정적 자극이 되었습니다. 우승팀 *BellKor's Pragmatic Chaos* 는 수백 개의 모델을 앙상블한 결과로 우승했고, 이 사례는 *대규모 데이터 + 정교한 알고리즘 = 큰 비즈니스 가치* 라는 패턴을 산업에 확산시켰습니다.

그러나 이 패러다임에는 결정적 약점이 있었습니다. *원시 데이터 (이미지 픽셀, 자연어 단어, 음성 파형)* 를 직접 입력으로 받지 못한다는 점입니다. 입력으로 사용하려면 인간이 *특성 공학 (Feature Engineering)* 을 통해 *의미 있는 특성* 을 미리 뽑아내야 했습니다.

예를 들어, 사진에서 고양이를 분류하려면 인간이 *귀의 모양·털의 색·눈의 형태* 같은 특성을 정의하고, 사진에서 그 특성들의 값을 추출하는 코드를 직접 만들어야 했습니다. 그 다음에야 SVM 같은 분류기가 학습할 수 있었습니다. 특성 공학은 도메인 전문성과 실험을 요구하는 어려운 일이었으며, *어떤 특성이 좋은가* 는 결국 인간의 직관에 의존했습니다.

이 시기의 컴퓨터 비전 연구자들은 *SIFT (1999)·HOG (2005)* 같은 *손으로 설계한 특성 추출기* 를 만드는 데 큰 노력을 기울였습니다. 이런 특성 추출기는 매우 정교하고 효과적이었지만, *새로운 도메인에 적용할 때마다 처음부터 설계* 해야 하는 한계를 가졌습니다. 자연어 처리에서도 *TF-IDF·n-gram·구문 트리* 같은 손으로 설계한 특성이 표준이었습니다.

### 2.3 표현 학습 시대 (2012 - 현재)

2012 년 ImageNet 대회에서 AlexNet 이라는 깊은 합성곱 신경망(CNN) 이 *기존 SOTA 를 큰 격차로 압도* 한 사건이 패러다임 전환의 출발점이 되었습니다.

ImageNet 대회 (정식 명칭: ImageNet Large Scale Visual Recognition Challenge, ILSVRC) 는 2010 년부터 시작된 컴퓨터 비전 분야의 가장 권위 있는 벤치마크였습니다. 100 만 장 이상의 라벨된 이미지에서 1000 개의 클래스를 분류하는 것이 과제였고, 매년 *상위 5 개 후보 안에 정답이 있는가* 를 측정하는 *Top-5 오류율* 을 줄이는 것이 목표였습니다.

2010-2011 년 우승팀들의 Top-5 오류율은 약 25-28% 였으며, 매년 약 1-2%p 씩 점진적으로 개선되고 있었습니다. 그런데 2012 년 Toronto 대학의 Alex Krizhevsky·Ilya Sutskever·Geoffrey Hinton 이 제출한 *AlexNet* 이 *Top-5 오류율 15.3%* 를 기록하며 2 등 (26.2%) 과 11%p 라는 압도적 차이로 우승했습니다.

이 사건의 의미는 단순히 *새 알고리즘이 더 좋았다* 가 아닙니다. 더 본질적으로, 이 사건은 *인간이 특성을 직접 설계하지 않아도, 깊은 신경망이 데이터로부터 좋은 표현을 자동으로 학습할 수 있다* 는 것을 증명했습니다. 이 패러다임을 *표현 학습 (Representation Learning)* 이라 부릅니다.

표현 학습 패러다임의 핵심:

- *원시 데이터* (픽셀·문자·파형) 를 그대로 입력합니다.
- *깊은 신경망의 중간 층* 들이 점차 더 추상적인 표현을 만들어 냅니다.
- *마지막 층* 이 그 표현을 사용해 최종 예측을 만듭니다.

CNN 의 경우, 첫 층은 *엣지·선* 같은 저수준 특성을 학습하고, 중간 층은 *눈·코·입* 같은 부품 수준 특성을, 마지막 층은 *얼굴·고양이* 같은 객체 수준 특성을 학습합니다. 이 모든 특성은 인간이 정의한 것이 아니라 *모델이 학습 데이터로부터 스스로 발견한* 것입니다.

흥미로운 사실은 *AlexNet 이 사용한 알고리즘 자체* 는 1980-90 년대에 이미 알려져 있던 합성곱 신경망과 크게 다르지 않았다는 점입니다. *Yann LeCun* 의 *LeNet* (1989, 1998) 이 이미 같은 아이디어를 사용했습니다. AlexNet 의 성공은 *알고리즘의 혁신* 이라기보다 *세 가지 조건의 동시 등장* 이 만든 결과였습니다.

1. **데이터의 폭발** — ImageNet 자체가 100 만 장 이상의 라벨된 이미지를 제공한 첫 대규모 데이터셋이었습니다.
2. **GPU 컴퓨팅의 보급** — NVIDIA 의 CUDA 가 2007 년 공개되어, 일반 연구자도 GPU 로 신경망을 학습시킬 수 있게 되었습니다. AlexNet 학습에는 GTX 580 GPU 두 장이 사용되었습니다.
3. **알고리즘 개선** — ReLU 활성화 함수, Dropout, 데이터 증강 같은 기법들이 정착하면서 깊은 신경망 학습이 안정화되었습니다.

같은 패러다임이 자연어(워드 임베딩 (2013)·BERT (2018)), 음성(wav2vec (2019)), 다중 모달(CLIP (2021)) 모든 영역으로 확장되었고, 현재 AI 의 거의 모든 발전은 이 표현 학습 패러다임 위에서 일어납니다.

### 2.4 파운데이션 모델 시대 (2018 - 현재)

표현 학습 시대 안에서, 더 좁고 더 강력한 흐름이 *파운데이션 모델* 입니다.

*파운데이션 모델* 은 *거대한 일반 데이터로 한번 사전학습한 모델을, 여러 다운스트림 작업에 미세조정하거나 그대로 적용해 사용* 하는 패러다임입니다. 2018 년 BERT 의 등장이 이 흐름의 본격적 시작점이며, 2020 년 GPT-3 가 *거대 규모* 의 효과를 증명하면서 패러다임이 확립되었습니다.

*파운데이션 모델* 이라는 용어 자체는 2021 년 8 월 Stanford HAI (Human-Centered AI Institute) 가 발표한 보고서 *On the Opportunities and Risks of Foundation Models* 에서 정식으로 명명되었습니다. 100 명 이상의 연구자가 공저자로 참여한 이 200 페이지 분량의 보고서는 *이 새로운 패러다임이 사회·경제·윤리 전반에 미치는 영향* 을 광범위하게 다뤘습니다.

이 패러다임의 핵심:

- 한 번의 *사전학습 (Pretraining)* 에 막대한 데이터·컴퓨팅·시간을 투입합니다.
- 그 결과로 만들어진 *파운데이션 모델* 은 *프롬프트·미세조정·RAG·도구 사용* 같은 가벼운 적응만으로 다양한 작업에 적용됩니다.
- 한 번의 사전학습 비용을 *수많은 응용 사례* 에 분산함으로써 전체 비용 효율을 극대화합니다.

이 패러다임이 현재 AI 엔지니어 직무가 등장한 직접적인 배경입니다. 이전 시대에는 *각 작업마다 모델을 따로 학습* 해야 했지만, 파운데이션 모델 시대에는 *외부에서 만들어진 거대 모델을 가져다 적용* 하는 일이 일상이 되었습니다. AI 엔지니어는 이 *적용* 영역의 전문가입니다.

### 2.5 패러다임 변천의 의미

지금까지 *규칙 기반 → 통계 기반 → 표현 학습 → 파운데이션 모델* 의 네 단계를 봤습니다. 이 변천은 단순한 기술 발전 이상의 의미를 가집니다.

각 단계는 *직전 단계가 풀지 못한 문제* 를 풀면서 등장했습니다.

- *규칙 기반* 은 *지식을 명시적으로 표현* 하는 데 강했지만 *암묵적 지식 추출* 에 약했음.
- *통계 기반* 은 *데이터에서 패턴 학습* 에 강했지만 *원시 데이터 처리·특성 공학* 에 약했음.
- *표현 학습* 은 *원시 데이터에서 좋은 표현 추출* 에 강했지만 *작업마다 모델을 따로 학습* 해야 하는 비효율이 있었음.
- *파운데이션 모델* 은 *한 모델을 다양한 작업에 적응* 시키는 데 강하지만, *학습 비용이 거대* 하고 *몇몇 큰 회사에 종속* 되는 새 문제를 만들었음.

이 패턴은 *기술 발전이 항상 trade-off* 라는 사실을 보여 줍니다. 새 패러다임이 옛 문제를 풀지만 새 문제를 만들어 냅니다. AI 엔지니어로서 이 사실을 인식하면, *현재 패러다임의 한계가 다음 패러다임의 씨앗* 이라는 사고를 할 수 있게 됩니다.

다음 패러다임은 *어떤 모습일까요*? 정답은 없지만, 현재 활발히 연구되는 후보들은 다음과 같습니다.

- *추론 시간 컴퓨팅 (Test-time Compute)* — o1 스타일의 *생각하는 시간* 을 늘리는 모델
- *세계 모델 (World Model)* — 단순한 패턴 학습이 아닌 *물리·인과 모델*
- *체화된 AI (Embodied AI)* — 로봇·자율주행 같은 *물리 세계와 상호작용* 하는 AI
- *효율화된 모델* — 작은 모델·디스틸레이션·MoE 로 *비용 감소*
- *멀티모달 통합* — 텍스트·이미지·음성·비디오를 *하나의 모델* 로

이 후보들 중 어느 것이 다음 패러다임이 될지는 5-10 년 후에 명확해질 것입니다.

### 2.6 챕터 정리

AI 의 패러다임은 *규칙 기반 → 통계 기반 → 표현 학습 → 파운데이션 모델* 의 네 단계를 거쳐 왔습니다. 각 패러다임은 이전의 본질적 한계를 극복하면서 등장했고, 동시에 새로운 한계를 만들었습니다. 현재 AI 엔지니어 직무는 가장 최근 단계인 *파운데이션 모델 시대* 의 특성을 깊이 반영하며, 이 시대 특유의 *학습/적응 분리* 가 직무를 가능하게 만든 직접적 동력입니다. 다음 챕터에서는 그 시대에 살고 있는 우리가 다루는 *현대 AI 시스템* 의 구성요소를 봅니다.

## 3. 현대 AI 시스템의 구성요소

### 3.1 모델은 부품일 뿐이다

처음 분야를 접하는 사람들이 가장 자주 빠지는 함정은 *AI 시스템 = 모델* 이라고 생각하는 것입니다. *AI 엔지니어가 한다는 일* 을 *모델을 만들고 띄우는 일* 정도로 좁게 보면, 실제 직무의 70-80% 를 차지하는 다른 활동을 놓치게 됩니다.

이 함정은 학습 자료의 편향에서 옵니다. 대부분의 AI 강의·교재는 *모델 자체의 구조·학습 과정* 을 다룹니다. 신경망 그림·역전파 수식·아키텍처 도식이 자료의 90% 를 차지합니다. 그러나 실제 산업 현장에서 AI 엔지니어가 *모델 자체에 시간을 쓰는 비율* 은 훨씬 낮습니다.

현대 AI 시스템은 *모델* 이라는 한 부품을 중심으로 *데이터·인프라·평가·배포·관측·거버넌스* 라는 여섯 부품이 함께 구성된 *복합 시스템* 입니다. 모델은 그 중 가장 *눈에 띄는* 부품일 뿐, *가장 시간이 많이 드는* 부품도, *가장 비용이 많이 드는* 부품도, *가장 자주 바뀌는* 부품도 아닙니다.

이 사실을 *시스템 사고* 로 받아들이는 것이 AI 엔지니어 직무의 핵심 정신입니다. 자동차로 비유하면, *엔진* 이 가장 눈에 띄는 부품이지만 *자동차의 가치는 엔진만으로 결정되지 않습니다*. 변속기·서스펜션·브레이크·차체 디자인·소프트웨어·정비 네트워크가 모두 결합되어야 자동차가 운전자에게 가치를 줍니다. AI 시스템도 마찬가지입니다.

### 3.2 7 가지 구성요소

현대 AI 시스템의 표준 구성요소를 다음 일곱으로 분해합니다.

```
+----------------------------------------------------------+
|  AI 시스템                                               |
|                                                          |
|  +-------------+     +-------------+    +-------------+  |
|  |  데이터     | ──→ |  학습/적응  | ──→|  평가       |  |
|  +-------------+     +-------------+    +-------------+  |
|        ↓                   ↓                  ↓         |
|        ↓             +-------------+          ↓         |
|        ↓             |  모델 산출물 |          ↓         |
|        ↓             +-------------+          ↓         |
|        ↓                   ↓                  ↓         |
|        ↓             +-------------+          ↓         |
|        └────────────→|  배포/서빙  |←─────────┘         |
|                      +-------------+                    |
|                            ↓                            |
|                      +-------------+                    |
|                      |  관측/모니터 |                    |
|                      +-------------+                    |
|                            ↓                            |
|                      +-------------+                    |
|                      |  거버넌스   |                    |
|                      +-------------+                    |
+----------------------------------------------------------+
```

각 요소를 자세히 봅니다.

**1. 데이터.** 모든 AI 시스템의 출발점입니다. 학습 데이터·평가 데이터·운영 시 사용자 입력·검색용 인덱스 데이터·피드백 데이터 등 다양한 형태로 존재합니다. 데이터의 *품질·다양성·시의성* 이 모델 품질의 상한선을 결정합니다. 데이터 자체는 *모델보다 오래 살아남으며* 더 큰 자산이 됩니다.

데이터에 대한 한 가지 깊은 통찰은 *Andrew Ng* 가 강조한 *Data-Centric AI* 운동입니다. 그는 산업 현장의 ML 프로젝트에서 *모델 개선보다 데이터 개선이 더 큰 효과* 를 만드는 경우가 압도적으로 많다고 주장했습니다. 같은 모델 코드라도 *학습 데이터의 라벨 정확도* 를 5%p 올리면 모델 정확도가 그보다 더 크게 올라가는 일이 흔합니다.

**2. 학습/적응.** 모델을 만들거나 기존 모델을 자기 도메인에 맞게 변형하는 단계입니다. 사전학습부터 미세조정·LoRA·프롬프트 설계까지의 모든 활동을 포함합니다. 이 단계의 결과물은 *모델 가중치 + 사용 가이드* 입니다.

파운데이션 모델 시대에는 *학습* 과 *적응* 이 명확히 분리됩니다. 학습은 외부 (또는 사내 ML 팀) 에서 한 번 일어나고, 적응은 AI 엔지니어가 자기 도메인·자기 사용 사례에 맞춰 반복합니다. 적응의 비중이 점점 커지고 있으며, 어떤 사용 사례에서는 적응만으로 충분히 좋은 결과를 만들 수 있습니다.

**3. 모델 산출물.** 학습/적응의 결과로 나오는 *가중치 파일·모델 카드·평가 결과* 의 묶음입니다. 표준 포맷(`safetensors`, `gguf`) 으로 저장되며, *버저닝* 과 *재현 가능성* 이 중요합니다.

모델 산출물의 핵심은 *모델 가중치 자체뿐 아니라 그 모델을 사용하는 데 필요한 모든 메타데이터* 가 함께 묶이는 것입니다. *모델 카드 (Model Card)* 라는 표준 양식이 2018 년경 Margaret Mitchell 등에 의해 제안되어, 현재 Hugging Face Hub 의 모든 모델이 모델 카드를 첨부하는 관행이 자리잡았습니다.

**4. 평가.** 모델이 *실제 사용자에게 의도한 행동을 하는가* 를 측정합니다. 자동 평가·인간 평가·LLM-as-Judge·도메인 특화 평가 등 다양한 형태가 있습니다. 평가는 *한 번* 이 아니라 *지속적으로* 일어나며, 모델이 시간이 지나도 여전히 의도한 행동을 하는지 검증합니다.

LLM 시대에 평가는 더 어려워졌습니다. 분류 모델은 *정답이 명확* 하지만, 생성 모델은 *정답이 여러 개* 입니다. *어떤 응답이 좋은가* 는 본질적으로 주관적이며, 자동 평가의 한계가 명확합니다. 따라서 *LLM-as-Judge*·*인간 평가*·*Arena (사용자 선호 비교)* 같은 새로운 평가 방식이 등장했습니다.

**5. 배포/서빙.** 모델을 *사용자가 호출 가능한 시스템* 으로 띄우는 단계입니다. vLLM·Triton·TGI 같은 서빙 프레임워크 위에 동적 배칭·KV 캐시·오토스케일을 구성합니다. 비용·지연·처리량·가용성의 트레이드오프가 결정됩니다.

서빙 영역은 *백엔드 엔지니어의 기존 역량이 가장 직접적으로 활용* 되는 영역입니다. API gateway·로드밸런서·자동 스케일·관측성 같은 도구는 일반 백엔드 시스템과 거의 같으며, *모델 특화 부품 (vLLM·KV 캐시)* 만 새로 익히면 됩니다.

**6. 관측/모니터.** 배포된 시스템의 *실시간 행동* 을 측정합니다. 지연·오류율 같은 시스템 지표뿐 아니라, *환각율·정확도 드리프트·비용 변화* 같은 모델 지표도 포함합니다. 관측 데이터는 다시 *데이터* 영역으로 환류해 다음 학습에 사용됩니다.

LLM 관측성은 비교적 새로운 영역이며, *Helicone·Langfuse·LangSmith* 같은 도구가 표준이 되어가고 있습니다. 일반 APM 도구 (Datadog·New Relic) 가 처리하는 *지연·오류율* 외에 *프롬프트·응답·토큰 수·비용* 같은 LLM 특화 정보를 함께 수집·분석하는 도구입니다.

**7. 거버넌스.** 시스템의 *책임 영역* 을 다룹니다. 안전·공정성·프라이버시·규제 준수·접근 제어·감사 가능성이 포함됩니다. 처음부터 잘 설계되지 않으면 나중에 *되돌리기 매우 어려운* 부분입니다.

EU AI Act (2024 통과) 와 미국 NIST AI Risk Management Framework 같은 규제·표준이 빠르게 형성되고 있어, 거버넌스의 중요성이 매년 커지고 있습니다. 특히 의료·금융·법률 같은 *고위험 영역* 에서는 거버넌스가 *기술 결정의 일부* 로 통합되어야 합니다.

### 3.3 데이터 흐름

이 일곱 요소 사이의 데이터 흐름을 보면 시스템의 *살아 있는 순환* 이 보입니다.

데이터 영역에서 출발한 데이터는 *학습/적응* 으로 흐르고, 거기서 *모델 산출물* 이 나옵니다. 모델 산출물은 *평가* 로 가서 검증되고, 통과하면 *배포/서빙* 으로 옮겨집니다. 배포된 시스템은 사용자 입력을 받아 응답을 만들며, 그 과정에서 *관측 데이터* 를 생성합니다. 관측 데이터는 다시 *데이터* 영역으로 들어가 다음 학습 사이클의 입력이 됩니다.

이 순환은 *데이터 플라이휠 (Data Flywheel)* 이라 불리며, 좋은 AI 제품과 평범한 AI 제품을 가르는 가장 큰 차이입니다. 한 바퀴가 돌 때마다 모델이 조금씩 더 좋아지고, 그 개선이 더 많은 사용자를 유치하고, 그 사용자가 다시 더 많은 데이터를 만듭니다.

데이터 플라이휠을 잘 운영하는 회사의 대표 사례:

- *Tesla 의 자율주행* — 도로에서 발생한 *엣지 케이스* 를 자동으로 수집해 다음 모델 학습에 활용
- *OpenAI 의 ChatGPT* — 사용자의 *👍/👎* 피드백을 RLHF 데이터로 사용
- *Spotify 의 추천* — 사용자의 *재생·스킵·좋아요* 행동을 다음 추천 모델의 입력으로

이 플라이휠이 *큰 회사의 AI 가 점점 더 강해지는 이유* 입니다. 작은 회사는 *처음부터 충분한 데이터* 가 없으므로 플라이휠이 천천히 돕니다. 따라서 작은 회사는 *데이터 효율적인 학습 (Few-shot, RAG, 전이 학습)* 같은 기법으로 데이터의 부족을 보완해야 합니다.

### 3.4 일곱 요소의 비중

회사·팀·제품 단계에 따라 일곱 요소의 비중은 크게 달라집니다.

초기 스타트업의 AI 시스템은 *학습/적응 + 배포/서빙* 에 비중이 쏠립니다. 평가·관측·거버넌스는 *최소한* 만 갖춰지고, 데이터는 *외부 모델 API* 가 학습한 것에 의존합니다. 이 시기의 AI 엔지니어는 *빠른 프로토타입* 에 집중합니다.

성숙한 제품의 AI 시스템은 *데이터 + 평가 + 관측 + 거버넌스* 의 비중이 크게 늘어납니다. 학습/적응과 배포는 *자동화* 되어 인간 시간을 적게 잡아먹고, 인간은 *어떤 데이터로 어떤 평가를 어떻게 통과시킬 것인가* 같은 정책 수준 의사결정에 시간을 씁니다.

자신이 향하는 회사·제품의 *어느 단계* 에 있는지를 파악하면, *어느 요소를 깊이 학습해야 하는지* 가 결정됩니다.

### 3.5 챕터 정리

현대 AI 시스템은 *모델 + 데이터 + 인프라 + 평가 + 배포 + 관측 + 거버넌스* 의 일곱 부품으로 구성되며, 데이터 플라이휠로 묶여 순환합니다. 모델은 일곱 부품 중 하나일 뿐이며, AI 엔지니어의 시간은 일곱 부품 모두에 분산됩니다. 회사 단계 (초기 vs 성숙) 에 따라 일곱 요소의 비중이 다르며, 자신이 향하는 단계의 요구를 미리 알면 학습 우선순위가 결정됩니다. 다음 챕터에서는 *어느 부품에 자원을 투자해야 가장 큰 결과가 나오는가* 를 결정하는 *모델·데이터·인프라의 삼각형* 을 봅니다.

## 4. 모델·데이터·인프라의 삼각형

### 4.1 세 자원의 트레이드오프

AI 시스템의 결과 품질은 결국 *세 가지 자원* 의 함수입니다.

- **모델** — 어떤 알고리즘·아키텍처·사이즈를 쓰는가
- **데이터** — 얼마나 많고 좋은 학습/평가/검색 데이터가 있는가
- **인프라** — 어떤 GPU·메모리·네트워크가 있는가

세 자원 중 하나에 무한 자원을 부어도 다른 둘이 부족하면 결과가 나오지 않습니다. 이 셋이 *균형* 을 이뤄야 시스템이 작동합니다.

이 삼각형 사고는 *Rich Sutton* 의 2019 년 에세이 *The Bitter Lesson* 에서 한 형태로 표현되었습니다. 그는 *AI 연구 70 년의 가장 큰 교훈* 이 *컴퓨팅 자원을 활용한 단순한 방법이 결국 인간의 지식을 활용한 정교한 방법을 이긴다* 는 것이라고 주장했습니다. 즉 *알고리즘의 정교함* 보다 *컴퓨팅 + 데이터의 규모* 가 결국 더 큰 효과를 만든다는 관찰입니다. 이 관찰은 GPT 시리즈의 성공을 잘 설명하며, 산업의 무게중심을 *모델 자체* 에서 *데이터 + 인프라* 로 옮긴 한 동력이기도 합니다.

### 4.2 같은 결과를 다른 비율로

흥미로운 점은, *같은 결과를 만들 때 세 자원의 비율을 다르게* 가져갈 수 있다는 점입니다. 어떤 회사는 모델에 큰 투자를 하고 데이터·인프라는 적게 쓰며, 다른 회사는 데이터에 큰 투자를 하고 모델은 외부 것을 쓰며, 또 다른 회사는 인프라에 투자해 *작은 모델을 빨리 많이* 돌립니다.

```
+--------+
|  모델   |
|        |
+--------+
   /  \
  /    \
+--------+--------+
|  데이터 |  인프라 |
|        |        |
+--------+--------+

이 삼각형 안의 어느 위치에 있는가가 회사의 전략
```

각 극단에 위치하는 전략을 봅니다.

**모델 중심 전략.** OpenAI·Anthropic·DeepMind 같은 회사가 여기에 위치합니다. 새로운 아키텍처·새로운 학습 알고리즘·새로운 정렬 기법 같은 *모델 자체의 혁신* 에 큰 자원을 투입합니다. 데이터와 인프라도 막대하지만, *차별점* 은 모델에 있습니다. 이 회사들이 *AGI 를 목표* 로 한다고 공언하는 이유는, 결국 *모델의 본질적 능력* 을 가장 멀리 밀고 가는 회사가 가장 큰 가치를 가진다는 가설 때문입니다.

**데이터 중심 전략.** 의료·금융·법률 같은 도메인 특화 회사가 여기에 위치합니다. 자기 도메인의 *고품질 라벨링 데이터*·*규제 통과 데이터* 가 핵심 자산이며, 모델은 외부에서 가져옵니다. *Andrew Ng* 가 주장하는 *Data-Centric AI* 운동도 이 전략의 한 형태입니다.

데이터 중심 전략의 좋은 사례는 *Bloomberg GPT* (2023) 입니다. Bloomberg 는 50 년간 축적한 금융 텍스트 데이터를 활용해 *50B 파라미터의 금융 도메인 LLM* 을 학습시켰습니다. 모델 아키텍처는 표준 Transformer 였지만, *데이터* 가 차별점이었습니다.

**인프라 중심 전략.** Cerebras·Groq·SambaNova 같은 하드웨어 회사, 또는 *추론 비용 최적화* 가 핵심 가치인 회사들이 여기에 위치합니다. *같은 모델을 더 빠르고 더 싸게* 돌리는 능력이 차별점입니다. *Groq 의 LPU (Language Processing Unit)* 는 같은 모델을 *NVIDIA GPU 보다 5-10 배 빠르게* 추론하는 것을 보여 주었고, 이것이 회사의 시장 가치를 만들고 있습니다.

대부분의 일반 기업은 세 극단의 중간 어딘가에 위치하며, *어느 방향으로 더 무게를 실을지* 가 전략적 선택입니다.

### 4.3 데이터의 누적 가치

세 자원 중 *시간이 갈수록 가장 큰 누적 가치* 를 가지는 것은 데이터입니다.

모델은 6 개월에서 1 년이면 더 좋은 모델로 대체됩니다. GPT-3.5 가 GPT-4 로, GPT-4 가 GPT-4-Turbo 로, 다시 GPT-4o 와 o1 으로 빠르게 대체되었습니다. 회사가 모델 자체에 투자한 자원은 *반감기* 가 짧습니다.

인프라도 빠르게 갱신됩니다. A100 이 H100 으로, H100 이 H200 과 B200 으로 대체됩니다. 한 세대의 GPU 가 시장의 표준으로 머무는 시간은 2-3 년에 불과합니다.

그러나 데이터는 *누적되며 사라지지 않습니다.* 사용자 행동 로그·도메인 특화 라벨·내부 문서·평가 셋·피드백 신호는 시간이 갈수록 더 많아지고 더 가치 있어집니다. *모델은 빌리고, 인프라는 임대하지만, 데이터는 자기 것으로 쌓는다* 는 전략이 장기적으로 가장 강력합니다.

### 4.4 균형 무너짐의 사례

세 자원의 균형이 무너졌을 때 어떤 일이 일어나는지를 보면 삼각형의 의미가 명확해집니다.

- *모델만 좋고 데이터가 부족* 한 경우: 거대한 GPT 급 모델을 가져와도 도메인 데이터가 없으면 *평범한 일반 답변* 만 만듭니다.
- *데이터만 많고 모델이 약함*: 좋은 데이터가 있어도 그 데이터를 잘 활용할 모델이 없으면 가치가 추출되지 못합니다.
- *모델·데이터는 좋은데 인프라가 부족*: 사용자 한 명에게 응답하는 데 30 초가 걸리면 제품으로 쓸 수 없습니다.
- *인프라만 풍부*: GPU 가 1000 장 있어도 *무엇에 쓸지* 가 명확하지 않으면 자원이 낭비됩니다.

AI 엔지니어로서 *현재 시스템의 가장 약한 자원이 무엇인가* 를 질문하는 습관이 이 삼각형 사고의 핵심입니다.

### 4.5 챕터 정리

AI 시스템의 결과는 *모델·데이터·인프라* 세 자원의 함수이며, 회사 전략은 어느 자원에 무게를 실을지로 표현됩니다. 세 자원 중 *데이터* 가 시간이 갈수록 가장 큰 누적 가치를 가집니다. AI 엔지니어는 *현재 시스템의 가장 약한 자원* 을 식별하고 거기에 자원을 집중하는 사고를 가져야 합니다. 다음 챕터에서는 이 자원들이 *학습 단계와 추론 단계* 에서 다르게 사용되는 이유를 봅니다.

## 5. 학습과 추론의 분리

### 5.1 두 단계의 본질적 차이

AI 시스템은 *학습 (Training)* 과 *추론 (Inference)* 이라는 두 단계로 명확히 분리됩니다. 두 단계는 *완전히 다른 워크로드* 이며, 다른 인프라·다른 도구·다른 SLA·다른 비용 모델을 가집니다.

**학습.** 모델 가중치를 *결정하는* 단계입니다.

- 한 번 일어남 (또는 주기적으로 재학습)
- 전체 학습 데이터를 *여러 번* 통과 (Epoch)
- 그래디언트 계산을 위한 *순전파 + 역전파* 모두 필요
- 메모리에 *모델 + 활성화 + 옵티마이저 상태* 를 모두 보관해야 함
- 시간 단위: 시간 ~ 일 ~ 주 ~ 개월
- 결과물: 가중치 파일

**추론.** 학습된 모델로 *예측을 만드는* 단계입니다.

- 사용자가 요청할 때마다 일어남 (지속적)
- 입력 한 건씩 처리 (또는 배치로 묶어 처리)
- *순전파만* 필요 (역전파 없음)
- 메모리에 *모델 가중치 + KV 캐시* 정도만 보관
- 시간 단위: 밀리초 ~ 초
- 결과물: 예측·응답

### 5.2 메모리 비용의 차이

두 단계의 메모리 비용 차이는 극적입니다.

7B 파라미터 모델을 예로 들면:

**학습 시 메모리 (FP16, AdamW):**
- 가중치: 7B × 2 bytes = 14 GB
- 그래디언트: 7B × 2 bytes = 14 GB
- AdamW 모멘트 1·2: 7B × 4 bytes × 2 = 56 GB
- 활성화 (순전파 결과): 시퀀스 길이·배치에 비례, 수십 GB
- **합계: 약 100 GB+**

**추론 시 메모리 (FP16):**
- 가중치: 14 GB
- KV 캐시: 시퀀스 길이·배치에 비례, 수 GB
- **합계: 약 15-20 GB**

학습은 *추론의 5-10 배* 메모리를 요구합니다. 같은 7B 모델도 학습은 H100 80GB 한 장으로 빠듯하지만, 추론은 RTX 4090 24GB 한 장으로 가능합니다.

이 메모리 비용 차이는 *학습은 큰 GPU 클러스터에서, 추론은 작은 GPU 한 장에서* 라는 산업 표준 패턴을 만든 직접적 원인입니다. 70B 모델 학습에는 보통 64-128 장의 H100 이 필요하지만, 같은 모델의 추론 (양자화 적용 시) 은 H100 한 장으로 가능합니다.

### 5.3 시간 단위의 차이

시간 단위도 극적으로 다릅니다.

학습은 *오프라인* 워크로드입니다. 며칠이 걸려도 결과만 좋으면 됩니다. 학습 도중에 GPU 가 잠깐 멈춰도 체크포인트에서 재개할 수 있습니다.

추론은 *온라인* 워크로드입니다. 사용자가 *지금 이 순간* 응답을 기다리고 있으며, 200 ms 가 지나면 사용자가 페이지를 떠납니다. 추론 시스템은 *고가용성·낮은 지연·자동 복구* 가 필수입니다.

이 시간 단위 차이가 *왜 같은 회사 안에서도 학습 인프라와 추론 인프라가 분리되어 운영되는지* 를 설명합니다. 학습 클러스터는 *처리량 최대화* 가 목표이고, 추론 클러스터는 *지연 최소화* 가 목표입니다. 두 목표는 종종 상충합니다.

### 5.4 비용 모델의 차이

비용도 다른 방식으로 누적됩니다.

학습은 *자본적 지출 (CapEx 성격)* 에 가깝습니다. 한 번에 큰 비용을 들여 모델을 만들고, 그 모델을 오래 사용합니다. 70B 모델을 사전학습하는 데 수백만 달러가 들지만, 그 모델은 수년간 활용될 수 있습니다.

추론은 *운영 비용 (OpEx 성격)* 입니다. 사용자 한 명·요청 한 건당 GPU 시간·토큰 수·전력에 비례하는 비용이 끊임없이 누적됩니다. 사용자 1 백만 명이 매일 사용하면 그 비용이 매월 수십만 달러가 됩니다.

따라서 *추론 비용 최적화* 가 AI 엔지니어의 가장 중요한 KPI 중 하나입니다. 학습 비용은 한 번이지만, 추론 비용은 *제품이 살아 있는 동안 영원히* 발생합니다.

### 5.5 AI 엔지니어의 영역

이 분리 위에서 *AI 엔지니어* 의 영역을 다시 정의할 수 있습니다.

> AI 엔지니어는 주로 *추론 단계* 에 시간을 쓰는 직무이며, 학습 단계의 결과물(모델) 을 *효율적으로 적용·운영* 하는 일을 합니다.

학습은 ML 엔지니어·AI 리서처·외부 모델 제공자(OpenAI 등) 의 영역입니다. AI 엔지니어가 학습을 다룬다면 그것은 *전체 사전학습이 아닌 가벼운 미세조정 (LoRA·QLoRA 등)* 에 한정됩니다. 그러나 *추론 영역의 깊이* 는 AI 엔지니어가 가장 강해야 할 부분입니다.

이 책의 Track K (GPU 와 추론 가속) 와 Track L (운영) 이 추론 단계의 깊이에 가장 큰 비중을 두는 이유가 여기에 있습니다.

### 5.6 챕터 정리

학습과 추론은 *메모리·시간·비용 모델·SLA 가 모두 다른 두 워크로드* 입니다. AI 엔지니어는 주로 추론 단계에 시간을 쓰며, 학습 결과물을 효율적으로 운영하는 일을 합니다. 추론 비용은 *제품이 살아 있는 동안 영원히 누적* 되므로, 추론 최적화가 가장 중요한 KPI 중 하나입니다. 다음 챕터에서는 *학습된 모델을 어디에서 가져오는가* 의 의사결정인 오픈소스 vs 폐쇄형 모델의 트레이드오프를 봅니다.

## 6. 오픈소스 vs 폐쇄형 모델

### 6.1 두 진영의 정의

**폐쇄형 모델 (Closed / Proprietary)** — 모델 가중치가 공개되지 않으며, *API 호출* 을 통해서만 사용할 수 있는 모델입니다.

대표 사례:
- OpenAI: GPT-4o, o1, o3
- Anthropic: Claude 3.5/3.7/4 series
- Google: Gemini 1.5/2.0 Pro/Flash

**오픈소스 모델 (Open / Open-Weight)** — 모델 가중치가 공개되어, 누구나 다운받아 자기 환경에서 실행하거나 재학습할 수 있는 모델입니다.

대표 사례:
- Meta: LLaMA 3, LLaMA 4
- Mistral: Mistral, Mixtral, Codestral
- Alibaba: Qwen 2/2.5/3
- DeepSeek: DeepSeek-V2/V3, DeepSeek-R1
- Microsoft: Phi-3/4

엄밀히는 *오픈 가중치 (Open-Weight)* 와 *완전 오픈 (Open-Source: 가중치 + 학습 코드 + 학습 데이터)* 이 구분되지만, 실무에서는 *가중치 공개* 를 기준으로 *오픈* 으로 묶어 부르는 경우가 많습니다.

이 구분이 중요한 이유는 *학습 코드와 데이터까지 공개* 된 모델은 *완전한 재현·검증* 이 가능하지만, *가중치만 공개* 된 모델은 *어떤 데이터로 학습되었는지 알 수 없는 블랙박스* 라는 점입니다. 후자는 *학습 데이터에 라이선스 위반·편향이 있어도 알 수 없다* 는 위험을 내포합니다.

### 6.2 트레이드오프 표

두 진영의 트레이드오프를 정리합니다.

```
+---------------------+----------------------+----------------------+
|  기준               |  폐쇄형 (API)        |  오픈 (자체 운영)   |
+---------------------+----------------------+----------------------+
|  초기 도입 비용     |  매우 낮음           |  높음 (인프라)      |
|  단위 비용          |  토큰당 가격         |  GPU 시간 + 전력    |
|  대규모에서 비용    |  매우 높음           |  낮음               |
|  지연               |  네트워크 + 모델     |  모델만             |
|  데이터 프라이버시  |  외부로 송출         |  내부 통제          |
|  최신 능력          |  최고 수준           |  6-12 개월 뒤따름   |
|  미세조정 자유도    |  제한적              |  완전               |
|  운영 부담          |  거의 없음           |  큼 (전담 인력)     |
|  벤더 종속          |  강함                |  약함               |
|  규제 준수          |  벤더 정책에 의존    |  자체 통제          |
+---------------------+----------------------+----------------------+
```

이 표에서 *어느 한 쪽이 절대적으로 우위* 인 항목은 없습니다. 회사의 상황과 우선순위에 따라 선택이 달라집니다.

### 6.3 폐쇄형이 유리한 상황

다음 상황에서는 폐쇄형이 우위입니다.

- *초기 프로토타입* 단계 — 빠르게 만들고 검증해야 하며, 인프라 구축 시간을 줄여야 함
- *최신 능력이 필요한 작업* — 가장 어려운 추론·코드 생성·멀티모달 작업
- *트래픽이 작거나 변동성이 큼* — 자체 운영 시 GPU 가 놀게 되는 시간이 김
- *미세조정 필요 없음* — 프롬프트와 RAG 만으로 충분
- *전담 ML 인력이 없음* — 운영 부담을 지기 어려움

대부분의 초기 스타트업과 *AI 가 핵심 가치가 아닌* 회사의 AI 부서는 폐쇄형으로 시작합니다.

### 6.4 오픈이 유리한 상황

다음 상황에서는 오픈이 우위입니다.

- *대규모 트래픽* — 토큰당 비용이 GPU 운영 비용을 크게 초과
- *데이터 프라이버시 민감* — 의료·금융·법률·국방 도메인
- *미세조정·LoRA 가 필요* — 도메인 특화·말투 조정·내부 지식 주입
- *지연이 매우 중요* — 외부 API 의 네트워크 지연이 SLA 를 위협
- *벤더 종속을 피하고 싶음* — 가격·정책 변경의 영향을 줄이고 싶음
- *오프라인 환경* — 인터넷이 안 되는 산업 현장·해외 격오지

성숙한 AI 제품·규제 산업·대량 트래픽 환경에서는 오픈이 점점 더 자연스러운 선택이 됩니다.

### 6.5 하이브리드 전략

실무에서는 *둘 중 하나만 쓰는 회사* 보다 *두 진영을 같이 쓰는 회사* 가 더 많습니다.

흔한 하이브리드 패턴:

- *기본 트래픽* 은 오픈 모델로 처리, *어려운 케이스* 만 폐쇄형 API 로 라우팅
- *프로토타이핑* 은 폐쇄형으로 빠르게, *프로덕션* 은 오픈으로 전환
- *내부 직원용* 은 폐쇄형 API, *고객 데이터 처리* 는 오픈 자체 운영
- *최신 모델 평가* 는 폐쇄형으로 시작, *충분히 좋아진 시점* 에 자체 학습된 오픈 모델로 전환

이 하이브리드 패턴을 잘 설계하는 것이 AI 엔지니어의 중요한 의사결정 영역이며, *모델 라우터·게이트웨이* 같은 도구가 이 패턴을 지원합니다.

### 6.6 의사결정 흐름

새 프로젝트에서 *오픈 vs 폐쇄형* 을 결정하는 표준 흐름을 정리합니다.

```
[시작] → 개념 검증 단계인가?
   YES → 폐쇄형 API 로 빠르게 시작 (1-2 주 안에 PoC)
   NO  → ↓

[프로덕션 단계] → 트래픽이 큰가? (예: 일 100 만 요청+)
   YES → 비용 비교: API 토큰 비용 vs GPU 운영 비용
        → API 비용 > GPU 비용 × 2  → 오픈 모델 자체 운영
        → 그 외                   → 폐쇄형 유지
   NO  → ↓

[데이터 민감도] → 외부 API 에 데이터를 보내도 되는가?
   NO  → 오픈 모델 자체 운영 필수
   YES → ↓

[도메인 특화도] → 미세조정이 필요한가?
   YES → 오픈 모델 (또는 폐쇄형 미세조정 API)
   NO  → ↓

[운영 역량] → 전담 ML 인프라 인력이 있는가?
   YES → 오픈 모델도 가능
   NO  → 폐쇄형 권장 (운영 부담 회피)
```

이 흐름이 모든 상황에 답을 주지는 않지만, *결정의 주요 분기점* 을 빠뜨리지 않게 도와줍니다.

### 6.7 챕터 정리

오픈과 폐쇄형 모델의 선택은 *비용·프라이버시·최신성·미세조정 자유도·운영 부담* 의 다섯 축에서 트레이드오프됩니다. 어느 한 쪽이 절대적 우위는 아니며, 실무에서는 *하이브리드 전략* 이 가장 흔합니다. AI 엔지니어는 두 진영의 장단을 모두 익혀, 상황에 맞는 선택을 할 수 있어야 합니다. 다음 챕터에서는 두 진영을 가능하게 만든 패러다임인 *파운데이션 모델* 을 더 자세히 봅니다.

## 7. 파운데이션 모델 시대

### 7.1 무엇이 바뀌었는가

*파운데이션 모델* 이라는 용어는 2021 년 스탠퍼드 HAI 의 보고서 *On the Opportunities and Risks of Foundation Models* 에서 정식으로 명명되었습니다. 그러나 그 패러다임 자체는 2018 년 BERT, 2019 년 GPT-2, 2020 년 GPT-3 의 흐름 속에서 이미 형성되고 있었습니다.

이 시대 이전과 이후를 가르는 가장 큰 변화는 다음 한 문장으로 요약됩니다.

> 이전: 작업마다 모델을 따로 학습한다.
> 이후: 한 거대 모델을 한 번 학습한 뒤, 모든 작업에 적응시킨다.

이전 시대에는 *번역 모델·요약 모델·분류 모델·NER 모델* 을 각각 따로 학습했습니다. 각 모델은 특정 작업에만 사용 가능했고, 새 작업이 등장하면 처음부터 새 모델을 만들어야 했습니다.

파운데이션 모델 시대에는 *하나의 거대 모델* 이 번역·요약·분류·NER·코드 생성·추론을 모두 *어느 정도 이상* 수준으로 수행합니다. 새 작업은 *프롬프트* 를 바꾸거나 *경량 미세조정* 을 통해 며칠 안에 추가할 수 있습니다.

이 변화는 단순한 기술 발전이 아닙니다. *AI 시스템 구축의 경제학 자체가 바뀐* 사건입니다. 이전 시대에는 *새 작업 = 새 데이터 수집 + 새 모델 학습 + 새 평가* 의 3-6 개월짜리 프로젝트였습니다. 파운데이션 모델 시대에는 *새 작업 = 새 프롬프트 + 평가* 의 며칠 짜리 프로젝트가 되었습니다. 이 100 배의 시간 단축이 *AI 엔지니어* 라는 직무가 등장한 직접적 배경입니다.

### 7.2 사전학습 + 적응의 두 단계

파운데이션 모델의 사용은 두 단계로 나뉩니다.

**사전학습 (Pretraining).** 거대한 일반 데이터(인터넷 텍스트·도서·코드·이미지) 로 모델을 처음부터 학습합니다. 이 단계는 막대한 컴퓨팅(수천-수만 GPU·수백만-수천만 달러) 이 필요하며, 보통 *모델 제공자* (OpenAI, Anthropic, Meta, Google 등) 가 담당합니다. 결과물은 *기반 능력* 을 가진 모델입니다.

GPT-3 의 사전학습 비용은 *약 460 만 달러* 로 추정됩니다 (Lambda Labs 분석). GPT-4 는 그보다 *수십 배 많은* 비용이 들었을 것으로 추정됩니다. 이 비용 규모 때문에 사전학습은 *몇 개의 큰 회사만 할 수 있는* 활동이 되었으며, 이는 *AI 의 권력 집중* 이라는 새 우려도 만들어 냈습니다.

**적응 (Adaptation).** 사전학습된 모델을 특정 작업·도메인·말투에 맞게 변형합니다. 이 단계는 비교적 가벼우며(단일 GPU·수 시간-수 일), AI 엔지니어와 작은 ML 팀이 담당합니다. 적응 방법은 다음과 같이 분류됩니다.

- **프롬프트 엔지니어링** — 모델 가중치를 바꾸지 않고 *입력 텍스트* 만 잘 설계
- **In-Context Learning** — 프롬프트에 *예시* 를 함께 넣어 *암묵적 학습*
- **RAG (Retrieval-Augmented Generation)** — 외부 데이터를 *검색해 컨텍스트로 주입*
- **미세조정 (Fine-Tuning)** — 작은 학습률로 *가중치를 약간 조정*
- **LoRA / PEFT** — *전체의 0.1-1% 만 학습* 하는 효율적 미세조정
- **RLHF / DPO** — 인간 선호 데이터로 *행동 정렬*

각 적응 방법은 비용·자유도·결과 품질이 다르며, 자세한 내용은 Track H (LLM) 와 Track M (미세조정) 에서 다룹니다.

### 7.3 능력의 일반성 (Generality)

파운데이션 모델의 결정적 특성은 *능력의 일반성* 입니다. 한 모델이 *다양한 종류의 작업* 을 어느 정도 수행할 수 있다는 점입니다.

GPT-3 가 처음 발표되었을 때 가장 충격적이었던 발견은 *모델이 학습 시점에 의도되지 않았던 작업* 들도 *프롬프트만으로* 수행할 수 있다는 사실이었습니다. 다음 토큰 예측만 학습한 모델이 산수·번역·코드 생성·역할극·요약을 *별도 미세조정 없이* 해냈습니다.

이 *창발 (Emergence)* 현상은 모델 크기가 일정 임계점을 넘으면 *능력이 갑자기 출현* 하는 것처럼 보이는 현상이며, 파운데이션 모델 시대를 *기존 ML 과 질적으로 다른 패러다임* 으로 만든 핵심입니다.

물론 모든 작업에서 같은 수준은 아닙니다. 어떤 작업은 *상용 서비스 수준* 으로 잘 하고, 어떤 작업은 *데모 수준* 만 하며, 어떤 작업은 *전혀 안 됩니다.* 어느 작업이 어느 수준에 속하는지를 빠르게 판별하는 능력이 AI 엔지니어의 핵심 역량 중 하나입니다.

### 7.4 멀티모달로의 확장

파운데이션 모델은 처음에는 텍스트 중심이었지만, 지금은 *멀티모달* 로 빠르게 확장되었습니다.

- *Vision-Language* — GPT-4V, Claude 3.5/4, Gemini, LLaVA, Qwen-VL
- *Vision-Language-Action* — RT-2, OpenVLA (로보틱스)
- *오디오* — Whisper (음성 인식), GPT-4o Voice, Gemini Live
- *비디오* — Sora, Veo, Runway Gen-3
- *3D* — TripoSR, Trellis

이 확장은 *같은 사전학습 + 적응 패러다임* 위에서 일어납니다. 모달리티가 다양해지더라도 AI 엔지니어가 다루는 인터페이스는 *모델 호출 + 컨텍스트 + 평가 + 운영* 으로 일관되며, 학습한 사고를 거의 그대로 이전할 수 있습니다.

### 7.5 시대가 만든 직무 변화

파운데이션 모델 시대는 *AI 엔지니어* 라는 직무를 *별도 직무* 로 만든 가장 큰 동력입니다.

- 이전 시대: ML 엔지니어 한 명이 *학습부터 배포까지 전부* 담당. 직무 분화가 약함.
- 파운데이션 모델 시대: *학습은 모델 제공자가, 적응·배포는 AI 엔지니어가* 담당. 직무 분리.

이 분리 덕분에 *수학·통계·연구 배경이 깊지 않아도* AI 시스템을 만들 수 있게 되었습니다. 일반 소프트웨어 엔지니어가 AI 엔지니어로 전직하기에 *역사상 가장 좋은 시기* 가 지금입니다.

다만 이 분리는 *영원하지 않을 수도* 있습니다. 시간이 지나 도구가 더 발전하면 *적응* 영역도 더 자동화되어 *프롬프트 엔지니어* 같은 더 가벼운 직무로 분화될 수 있습니다. 또는 *오픈 모델의 자체 학습* 이 점점 쉬워지면 적응과 학습의 경계가 다시 흐려질 수 있습니다. AI 엔지니어는 자신의 직무가 *어떻게 진화할지* 를 늘 의식해야 합니다.

### 7.6 챕터 정리

파운데이션 모델 시대는 *작업마다 모델을 만들던 패러다임* 을 *한 모델을 모든 작업에 적응시키는 패러다임* 으로 바꿨습니다. 사전학습은 외부에서, 적응은 AI 엔지니어가 담당하며, 이 분리가 AI 엔지니어 직무를 가능하게 한 직접적 동력입니다. 능력의 일반성·창발·멀티모달 확장이 이 시대의 핵심 특성입니다. 다음 챕터에서는 이 시대의 AI 엔지니어가 매일 마주치는 의사결정의 종류를 정리합니다.

## 8. AI 엔지니어가 매일 마주치는 의사결정

### 8.1 의사결정의 분류

이 권의 마지막 챕터로, 지금까지의 내용을 *AI 엔지니어가 매일 마주치는 구체적 질문* 으로 종합합니다. 이 질문들은 이후 100 권의 학습 자료가 답하려는 *현실의 질문* 들입니다.

의사결정을 다섯 범주로 나눕니다.

1. **모델 선택 결정**
2. **적응 전략 결정**
3. **데이터 결정**
4. **인프라·서빙 결정**
5. **운영·평가 결정**

### 8.2 모델 선택 결정

매일 마주치는 질문:

- 이 작업에 GPT-4o 를 쓸지 Claude 3.7 을 쓸지 Gemini 2.5 Pro 를 쓸지?
- 폐쇄형 API 를 쓸지 오픈 모델을 자체 호스팅 할지?
- LLaMA 3 8B 로 충분한가, 70B 가 필요한가?
- 일반 LLM 으로 풀지 도메인 특화 모델로 풀지?
- 단일 모델로 풀지 모델 캐스케이딩(Cheap → Expensive) 으로 풀지?
- 새로 발표된 모델로 갈아탈 가치가 있는가?

이 질문들의 답은 *비용·정확도·지연·라이선스·데이터 프라이버시* 의 다섯 축 비교로 만들어집니다. 한 번의 결정으로 끝나지 않고, *3-6 개월마다 재검토* 됩니다. 새 모델이 끊임없이 등장하기 때문입니다.

### 8.3 적응 전략 결정

매일 마주치는 질문:

- 프롬프트만으로 충분한가, 미세조정이 필요한가?
- RAG 로 풀 것인가, 미세조정으로 풀 것인가, 둘을 결합할 것인가?
- LoRA 가 적절한가, Full Fine-Tuning 이 필요한가, QLoRA 로 메모리를 줄일까?
- Few-shot 으로 충분한가, Chain-of-Thought 가 필요한가, Self-Consistency 까지 가야 하는가?
- 시스템 프롬프트를 어떻게 구조화할 것인가? 마크다운? XML 태그? 함수 호출?

적응 전략은 *데이터 양·도메인 깊이·정확도 요구·갱신 주기* 에 따라 결정됩니다. 일반적으로 다음 우선순위로 시도합니다.

1. *프롬프트 엔지니어링* (가장 가벼움) → 충분하면 여기서 멈춤
2. *RAG* (외부 지식이 필요한 경우)
3. *Few-shot / In-Context Learning*
4. *Light 미세조정 (LoRA)*
5. *Full 미세조정* (가장 무거움)

### 8.4 데이터 결정

매일 마주치는 질문:

- 평가 셋을 어떻게 구축할 것인가? 몇 건이 적절한가?
- 사용자 피드백을 어떻게 수집·정리·활용할 것인가?
- 합성 데이터를 사용할 것인가? 어디까지 안전한가?
- PII 를 어떻게 처리할 것인가? 익명화·삭제·통제 접근?
- 학습 데이터의 중복·오염을 어떻게 감지할 것인가?
- 도메인 데이터의 라벨링을 자동·반자동·인간 중 어떤 비율로 할 것인가?

데이터 결정은 *시간이 갈수록 누적되는 자산* 을 만드는 일이므로, *한 번의 결정이 수년간 영향* 을 줍니다.

### 8.5 인프라·서빙 결정

매일 마주치는 질문:

- vLLM 을 쓸지 Triton 을 쓸지 TGI 를 쓸지?
- GPU 한 장에 몇 명까지 동시에 받을 수 있는가? 동적 배칭의 최적 사이즈는?
- KV 캐시 메모리가 폭발하지 않도록 어떻게 통제할 것인가?
- 양자화(FP8, INT8, GPTQ, AWQ) 중 무엇을 쓸 것인가?
- `torch.compile` 또는 TensorRT-LLM 으로 컴파일할 가치가 있는가?
- 멀티 GPU 분산 추론(텐서 병렬·파이프라인 병렬) 이 필요한가?
- 오토스케일 임계값을 어떻게 설정할 것인가?

이 결정들은 *사용자가 직접 체감하는 응답 속도와 회사가 매월 지불하는 비용* 을 결정합니다.

### 8.6 운영·평가 결정

매일 마주치는 질문:

- 어떤 지표로 모델 성능을 모니터링할 것인가? (정확도? 환각율? 사용자 만족도?)
- 드리프트를 어떻게 감지할 것인가? 임계값은?
- 새 모델 버전을 어떻게 롤아웃할 것인가? 카나리·섀도우·전체?
- 인시던트가 발생하면 어떻게 롤백할 것인가?
- 사용자 피드백을 자동 평가에 어떻게 통합할 것인가?
- 안전·공정성·프라이버시 규제를 어떻게 검증할 것인가?

이 결정들은 *제품이 살아 있는 동안 지속되는* 일이며, AI 엔지니어가 *유지하고 발전시키는* 시스템의 가장 큰 부분입니다.

### 8.7 의사결정 표를 만드는 습관

위 다섯 범주의 의사결정에는 한 가지 공통 도구가 있습니다. *비교 표* 를 만드는 습관입니다.

좋은 AI 엔지니어는 결정을 내리기 전에 다음 형태의 표를 만듭니다.

```
+----------+-------+-------+-------+-------+
| 옵션     | 비용  | 지연  | 정확도| 위험  |
+----------+-------+-------+-------+-------+
| A        | 낮음  | 100ms | 85%   | 작음  |
| B        | 중간  | 200ms | 92%   | 중간  |
| C        | 높음  | 50ms  | 96%   | 큼    |
+----------+-------+-------+-------+-------+
```

표를 만든 뒤, 자기 작업의 *우선순위 (비용 vs 정확도 vs 지연)* 에 따라 옵션을 선택합니다. 표가 있으면 결정 근거가 *문서로 남으며*, 다른 사람과 토론할 때 *근거 기반 대화* 가 가능해집니다.

이 습관은 이 책의 거의 모든 권에서 반복적으로 등장하며, AI 엔지니어의 가장 중요한 사고 도구입니다.

### 8.8 챕터 정리

AI 엔지니어가 매일 마주치는 의사결정은 *모델 선택·적응 전략·데이터·인프라·운영* 의 다섯 범주로 분류되며, 각 범주는 *비교 표* 라는 공통 도구로 다뤄집니다. 이 책의 100 여 권은 결국 이 다섯 범주의 의사결정을 *근거 기반* 으로 내릴 수 있는 능력을 만드는 자료입니다. 다음 권부터는 이 의사결정에 필요한 *실제 도구와 환경* 을 갖추는 단계로 들어갑니다.

## 권 정리

이 권에서 우리는 다음을 배웠습니다.

- AI·ML·DL 은 *동의어가 아니라 포함 관계* 이며, 현재 산업의 무게중심은 가장 안쪽의 DL, 그중에서도 *파운데이션 모델* 에 있습니다.
- AI 의 패러다임은 *규칙 기반 → 통계 기반 → 표현 학습 → 파운데이션 모델* 의 네 단계를 거쳐 왔습니다.
- 현대 AI 시스템은 *모델·데이터·인프라·평가·배포·관측·거버넌스* 의 일곱 부품으로 구성되며 *데이터 플라이휠* 로 순환합니다.
- 시스템 결과는 *모델·데이터·인프라* 세 자원의 함수이며, 데이터가 시간이 갈수록 가장 큰 누적 가치를 가집니다.
- 학습과 추론은 *메모리·시간·비용 모델 모두 다른 두 워크로드* 이며, AI 엔지니어는 주로 추론 단계를 다룹니다.
- 오픈과 폐쇄형 모델 선택은 *비용·프라이버시·최신성·미세조정 자유도·운영 부담* 의 다섯 축에서 트레이드오프됩니다.
- 파운데이션 모델 시대는 *학습과 적응의 분리* 를 만들었고, 이 분리가 AI 엔지니어 직무를 가능하게 한 직접적 동력입니다.
- AI 엔지니어가 매일 마주치는 의사결정은 *모델 선택·적응·데이터·인프라·운영* 의 다섯 범주로 분류되며, *비교 표* 라는 공통 도구로 다뤄집니다.

가장 기억에 남겨야 할 한 줄은 **"AI 시스템은 모델 한 부품이 아니라 일곱 부품의 시스템이며, AI 엔지니어는 그 일곱 부품의 균형을 결정한다."** 입니다.

다음 권은 [Volume 3 — 도구 체인과 작업 환경](./volume_03_toolchain.md) 입니다. 거기서는 AI 엔지니어가 매일 사용하는 도구의 전체 지도를 그리고, 자기 노트북에 PyTorch·CUDA·Hugging Face 환경을 직접 구축합니다.

## 자가점검 질문

다음 질문에 막힘없이 답할 수 있을 때 다음 권으로 넘어가십시오.

1. AI·ML·DL 의 포함 관계를 그림 한 장으로 그리고, 각 영역에 속하는 알고리즘·모델 사례를 두 개씩 적으십시오.
2. *표현 학습* 패러다임이 이전의 *통계 기반 머신러닝* 과 다른 본질적 차이는 무엇입니까?
3. 현대 AI 시스템의 일곱 구성요소를 모두 나열하고, 각 요소가 *데이터 플라이휠* 안에서 어떻게 연결되는지 설명하십시오.
4. *모델·데이터·인프라* 의 삼각형에서, *데이터* 가 가장 큰 누적 가치를 가진다고 말하는 이유는 무엇입니까?
5. 학습과 추론의 *메모리 비용* 차이를 7B 모델 예시로 비교하십시오.
6. 폐쇄형 모델과 오픈 모델 중, *대규모 트래픽 + 데이터 프라이버시 민감 + 자체 ML 인력 보유* 환경에서는 어느 쪽이 적합합니까? 그 이유는?
7. 파운데이션 모델 시대가 *AI 엔지니어 직무* 를 가능하게 한 직접적 메커니즘을 한 문단으로 설명하십시오.
8. *Rich Sutton 의 The Bitter Lesson* 의 주장을 자기 말로 한 문단 적으십시오.
9. AI 엔지니어가 매일 마주치는 다섯 범주의 의사결정 중, 자신의 직무에서 가장 중요할 범주를 골라 그 이유를 설명하십시오.
10. 다음 패러다임의 후보 다섯 가지 중 자신이 가장 가능성이 크다고 보는 후보를 골라 그 이유를 적으십시오.
