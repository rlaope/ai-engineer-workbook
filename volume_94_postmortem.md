# Volume 94 — AI 인시던트 사후 분석

> 이 권이 끝나면 *모델이 잘못된 결정을 내려 사용자가 피해를 입은 사고* 가 발생했을 때 표준 절차로 분석하고 재발을 막을 수 있게 됩니다.

## 목적

AI 시스템 인시던트는 *일반 SW 인시던트와 다른 분석 사고* 가 필요합니다. 모델·데이터·프롬프트·인프라 어디에서든 원인이 있을 수 있고, *결정론적 재현* 이 어렵습니다. 표준 사후 분석 양식과 *AI 특화 5 Whys* 를 다집니다.

## 1. AI 인시던트의 종류

- *환각* — 사용자에게 잘못된 정보 제공
- *유해 응답* — 안전 가드레일 우회
- *비용 폭증* — 의도하지 않은 API 호출
- *지연 폭증* — SLA 위반
- *프라이버시 누출* — PII 노출
- *편향 결정* — 특정 집단에 불리

각 종류별로 *대응이 다름*.

---

## 2. 표준 6 섹션

```markdown
# Postmortem: {제목}

## 1. 요약 (Summary)
- 무엇이 일어났는가 (1-2 문장)
- 영향 범위 (사용자 수·금액·시간)

## 2. 타임라인
- 14:00 — 사용자 신고 시작
- 14:15 — 온콜 호출
- 14:30 — 원인 식별
- 14:45 — 임시 대응 (롤백)
- 15:30 — 영구 수정 배포

## 3. 영향 (Impact)
- 영향받은 사용자: 12,000 명
- 잘못된 응답: 500 건
- 비용 손실: $X
- 평판 영향

## 4. 근본 원인 (Root Cause)
- 직접 원인: 새 모델 배포 후 환각율 급증
- 근본 원인: 평가 셋이 새 모델의 약점을 잡지 못함

## 5. 대응 (Response)
- 잘못된 응답: 옛 모델로 롤백
- 사용자: 영향받은 사용자에게 사과·보상
- 모니터링: 환각 감지 알람 추가

## 6. 행동 항목 (Action Items)
- [ ] 평가 셋에 X 카테고리 100 건 추가 (담당: @alice, 1 주)
- [ ] 환각율 자동 모니터링 (담당: @bob, 2 주)
- [ ] 모델 배포 전 새 셋으로 게이트 통과 의무 (담당: @charlie, 즉시)
```

## 3. Blameless Postmortem

### 3.1 원칙

*개인 비난 없음*. 시스템·절차의 문제로 다룸.

이유: 비난 문화는 *사람이 사고를 숨김* → 학습 기회 손실.

### 3.2 구체적 표현

- ❌ "Alice 가 잘못된 모델을 배포했다"
- ✅ "배포 절차에 자동 검증 게이트가 없었다"

---

## 4. AI 특화 5 Whys

```
사용자가 잘못된 의료 조언을 받았다
  Why? 모델이 환각을 만들었다
  Why? 학습 데이터에 의료 영역이 약했다
  Why? 평가 셋에 의료 케이스가 없었다
  Why? 평가 셋 구축 시 도메인 다양성이 우선순위가 아니었다
  Why? 평가 표준 가이드라인이 없었다
```

각 Why 가 *더 깊은 시스템 문제* 를 드러냄. 5 단계가 끝나면 *근본적 개선* 이 보임.

---

## 5. 영향 정량화

가능하면 *수치로*:

- *영향받은 사용자 수*
- *잘못된 응답 비율*
- *비용 손실 (₩)*
- *복구 시간 (분/시간)*
- *평판 영향 (소셜 미디어 언급 수 등)*

수치는 *우선순위·자원 할당* 의 근거.

---

## 6. 재발 방지

### 6.1 행동 항목 양식

```
- [ ] 무엇을 (구체적)
- 왜 (이 행동이 인시던트와 어떻게 연결)
- 누가 (담당자)
- 언제까지 (기한)
- 어떻게 측정 (완료 기준)
```

### 6.2 추적

행동 항목은 *주간 회의* 에서 추적. 미완료 시 사유 기록.

---

## 7. 공유

사내 *모든 엔지니어가 읽을 수 있게* 공개. 비슷한 사고를 *다른 팀이 미리 방지*.
