## 목적

같은 작업을 *7B·70B·400B 모델 어느 것* 으로 풀어야 할까요? 큰 모델은 정확하지만 비싸고 느림, 작은 모델은 빠르고 싸지만 능력 한계가 있습니다. *모델 캐스케이딩·라우팅·증류* 같은 패턴이 비용과 정확도의 균형을 만듭니다.


## 3. 모델 캐스케이딩

### 3.1 발상

쉬운 쿼리는 *작은 모델로 처리*, 어려운 쿼리만 *큰 모델로 escalate*.

```
사용자 쿼리 → 작은 모델 → 응답
                  ↓
              자신감 낮음? → 큰 모델 → 응답
```

전체 비용 *크게 감소*. 정확도 *큰 모델만 사용한 경우와 비슷*.

### 3.2 자신감 측정

- *모델 출력의 logprob* 이 낮으면 자신감 낮음
- *답변에 "I'm not sure"* 같은 표현
- *작은 모델 vs 다른 작은 모델* 일치도

---

## 4. 모델 라우터

### 4.1 발상

쿼리의 *종류* 에 따라 *다른 모델로 라우팅*.

```python
def route(query):
    intent = classifier(query)
    if intent == "simple_qa":
        return small_model.chat(query)
    elif intent == "complex_reasoning":
        return large_model.chat(query)
    elif intent == "code_generation":
        return code_model.chat(query)
```

### 4.2 도구

LiteLLM, Portkey 같은 *AI Gateway* 가 라우팅을 표준화.

---

## 5. 작은 모델로 충분한 작업

대부분의 *분류·추출·라벨링·재작성·번역* 작업은 *7-8B 로 충분*. 큰 모델이 필요한 경우는:

- 다단계 추론·수학·코드 생성
- 긴 문서 이해
- 다국어 (특히 저자원 언어)
- 미묘한 뉘앙스·창의성

이 분류를 미리 해 두면 *모델 선택이 빠름*.

---

## 6. 증류 (Distillation)

큰 모델의 출력을 *작은 모델이 모방*. 비용을 크게 줄이면서 정확도 비슷.

```
GPT-4 (Teacher) → 100K 응답 생성 → LLaMA 3 8B 미세조정
```

자기 도메인 특화 시 매우 효과적.

---

## 권 정리

- 사이즈별 능력 사다리: 1-3B → 7-8B → 13B → 70B → 400B
- 비용 차이 10-100 배
- 모델 캐스케이딩 = 쉬움→작음, 어려움→큼
- 모델 라우터 = 의도별 다른 모델
- 증류 = 큰 모델 → 작은 모델

가장 기억할 한 줄: **"항상 가장 큰 모델을 쓰는 것은 비싼 게으름이며, 라우팅·캐스케이딩·증류로 같은 결과를 1/10 비용에 달성할 수 있다."**
