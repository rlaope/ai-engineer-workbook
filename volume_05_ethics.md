## 이 권을 읽기 전에

이 권은 *추상적 윤리 강의* 가 아닙니다. AI 엔지니어가 매일 마주치는 *구체적 결정* — 어떤 데이터를 학습에 쓸지, 어떤 출력을 사용자에게 보일지, 어떤 사용자에게 어떤 응답을 줄지 — 의 *판단 기준* 을 다룹니다.

윤리는 *코드 외 영역* 처럼 보이지만, 본질은 *시스템 설계의 일부* 입니다. 잘못된 데이터·잘못된 출력·잘못된 사용자 분류는 모두 *기술 결정* 에서 비롯되며, 그 결과는 *기술 외 영역의 피해* 로 이어집니다. 이 인과를 인식하는 것이 AI 엔지니어 윤리의 출발점입니다.

이 권은 *법무 자문이 아닙니다*. 실제 법적 판단이 필요한 상황에서는 회사의 법무팀·외부 전문가의 자문을 구해야 합니다. 이 권은 *법무팀과의 대화에서 더 정확한 질문을 던질 수 있게* 만드는 것을 목표로 합니다.

## 1. AI 시스템이 만들 수 있는 5 가지 해

### 1.1 차별과 편향

학습 데이터에 *역사적 차별* 이 반영되어 있으면, 모델이 그 차별을 *재생산하거나 증폭* 합니다.

흔한 사례:

- 이력서 선별 모델이 특정 성별·인종에 불리하게 학습
- 신용 평가 모델이 특정 지역 거주자를 자동으로 낮게 평가
- 의료 진단 모델이 특정 인구 집단에 정확도가 낮음
- 얼굴 인식이 특정 피부톤에서 오류율이 높음

이 사례들은 모두 *학습 데이터의 분포가 편향* 되어 있어 발생합니다. 모델이 *공정* 한 것이 아니라, *학습 데이터에 있던 편향을 그대로 학습* 한 결과입니다.

방어책:

1. *학습 데이터의 인구학적 분포를 측정·기록*
2. *집단별 정확도 격차 모니터링* — 표준 메트릭은 *Equal Opportunity, Demographic Parity* 등
3. *문제 집단에 대한 추가 학습 데이터 수집*
4. *결정의 인간 검토 단계 도입*

### 1.2 프라이버시 침해

AI 시스템은 종종 *민감한 개인 정보* 를 처리합니다.

- 의료 기록·진단 정보
- 금융 거래·소득
- 위치·이동 패턴
- 개인 메시지·이메일·통화

이 정보가 *유출되거나 의도하지 않게 추론* 되면 사용자에게 큰 피해가 발생합니다. 또한 LLM 의 경우 *학습 데이터의 개인 정보가 출력에 그대로 등장* 하는 *멤버십 추론 공격* 이 가능합니다.

방어책:

1. *수집 단계에서 PII 최소화* — 필요 없는 정보는 처음부터 수집하지 않음
2. *익명화·가명화 적용*
3. *접근 제어와 감사 로그*
4. *학습 데이터에서 PII 제거 또는 토큰 마스킹*
5. *사용자 동의의 명시적 획득과 철회 메커니즘*

### 1.3 잘못된 정보 (환각·허위 사실)

LLM 은 *그럴듯하지만 사실이 아닌* 답을 만들 수 있습니다 (환각). 사용자가 이 답을 사실로 받아들이면 의료·법률·금융 같은 영역에서 큰 피해가 발생합니다.

방어책:

1. *출처 인용 의무화* — RAG 로 검증 가능한 출처 함께 제공
2. *Confidence 표시* — 모델이 *불확실* 한 응답에 명시적 표시
3. *고위험 영역 (의료·법률) 차단 또는 강한 면책 조항*
4. *출력 검증 미들웨어* — 사실 확인 가능한 주장은 자동 검증

### 1.4 악용 가능성

AI 시스템은 *의도와 다르게 악용* 될 수 있습니다.

- LLM 으로 *피싱 이메일·사기 메시지* 대량 생성
- 이미지 생성 모델로 *가짜 신원·증거 사진* 제작
- 음성 합성으로 *보이스 피싱·딥페이크*
- 코드 생성으로 *맬웨어 작성*

방어책:

1. *Use Policy 명시* — 약관에 금지 사용 사례 적시
2. *입력 모더레이션* — 명백히 악용 의도인 요청 차단
3. *출력 워터마킹* — AI 생성물 식별 가능
4. *남용 모니터링과 계정 정지*

### 1.5 자동화로 인한 일자리 영향

AI 시스템이 사람의 일을 대체하면 *해고·임금 하락* 이 발생합니다. 이는 윤리 영역이지만, 동시에 *사회적·정치적* 영역이기도 합니다.

방어책 (조직 차원):

1. *변화 영향 영향 평가* — 어떤 직무가 어떻게 영향받는지 분석
2. *재교육·직무 전환 프로그램*
3. *생산성 증가의 노동자 공유*
4. *AI 의 보조 도구로 사용 (Augmentation) vs 대체 (Replacement) 의 의식적 선택*

이 영역은 한 AI 엔지니어 개인이 해결할 수 없지만, *결정의 영향을 인식하는 것* 부터가 출발점입니다.

### 1.6 챕터 정리

AI 시스템이 만들 수 있는 주요 해는 *차별·프라이버시 침해·잘못된 정보·악용·일자리 영향* 의 다섯입니다. 각 해는 *기술적 방어책* 이 있지만, 완전한 해결은 어렵습니다. AI 엔지니어는 *해의 가능성을 인식* 하고 *방어책을 의식적으로 도입* 해야 합니다.

## 2. 편향과 공정성 점검

### 2.1 편향의 원천

ML 모델의 편향은 다음 단계 어디에서나 들어올 수 있습니다.

1. **데이터 수집** — 특정 집단이 과소·과다 표현됨
2. **라벨링** — 인간 라벨러의 무의식적 편향
3. **특성 선택** — 보호 변수와 강하게 상관된 특성 사용
4. **모델 아키텍처** — 다수 집단에 최적화된 구조
5. **평가** — 다수 집단으로만 평가
6. **배포** — 의도하지 않은 사용자 집단에 적용

모델 자체는 *수학적 도구* 일 뿐이며, 편향은 거의 항상 *데이터와 운영 환경* 에서 들어옵니다.

### 2.2 공정성 메트릭

학술 분야에서 사용되는 공정성 메트릭 몇 가지:

- **Demographic Parity** — 보호 집단별 *양성 예측 비율* 이 같음
- **Equal Opportunity** — 보호 집단별 *True Positive Rate* 가 같음
- **Equalized Odds** — TPR 과 FPR 모두 같음
- **Calibration** — 보호 집단별 예측 확률의 정확도가 같음

이 메트릭들은 *서로 충돌* 할 수 있습니다. 예를 들어 *Demographic Parity 와 Calibration 을 동시에 만족하는 것은 일반적으로 불가능* (Chouldechova, 2017). 따라서 *어느 공정성을 우선* 할지의 *가치 판단* 이 필요합니다.

### 2.3 실무 점검 체크리스트

```
□ 학습 데이터의 인구학적 분포 측정·기록
□ 집단별 정확도·오류율 분석
□ 보호 변수 (성별·인종·나이·지역) 의 직간접 사용 점검
□ 새 모델 배포 전 공정성 메트릭 평가
□ 사용자 피드백 채널 (편향 신고)
□ 정기적 (분기) 공정성 감사
```

이 체크리스트가 *팀의 표준 절차* 가 되어야 편향을 사전에 잡을 수 있습니다.

### 2.4 챕터 정리

편향은 *데이터·라벨·특성·아키텍처·평가·배포* 단계 어디에서나 들어올 수 있습니다. 공정성 메트릭은 *서로 충돌* 할 수 있어 가치 판단이 필요하며, 표준 점검 체크리스트로 *조직 절차에 통합* 합니다.

## 3. 프라이버시와 데이터 거버넌스

### 3.1 PII (개인 식별 정보) 의 종류

직접 식별 가능:
- 이름·주민등록번호·여권번호·운전면허번호
- 이메일·전화번호·주소
- 얼굴·지문·홍채

간접 식별 가능 (조합 시):
- 생년월일 + 우편번호 + 성별 (87% 의 미국인 식별 가능, Sweeney 2000)
- 위치 이력
- 검색 기록

후자는 *개별로는 식별이 안 되지만 조합 시 식별 가능* 한 정보입니다. AI 시스템은 종종 이런 조합을 *의도하지 않게* 생성합니다.

### 3.2 표준 보호 기법

- **익명화 (Anonymization)** — 직접 식별자 제거. 그러나 *재식별 공격* 가능.
- **가명화 (Pseudonymization)** — 식별자를 다른 ID 로 치환. 키 저장이 따로 됨.
- **k-anonymity** — 같은 속성 조합에 *최소 k 명* 이 매핑되도록.
- **차등 프라이버시 (Differential Privacy, DP)** — 통계적 노이즈로 *개별 추론 불가능*.
- **연합 학습 (Federated Learning)** — 데이터를 모으지 않고 *모델만 모음*.
- **동형 암호 (Homomorphic Encryption)** — 암호화된 상태에서 연산.

이 기법들은 *적용 비용과 보호 강도의 트레이드오프* 가 있습니다. 도메인의 위험도에 맞춰 선택합니다.

### 3.3 GDPR·CCPA 의 핵심

EU GDPR (2018) 과 미국 CCPA (2020) 같은 규제는 다음 권리를 사용자에게 부여합니다.

- *접근 권리* — 회사가 보유한 자기 정보를 볼 권리
- *수정 권리* — 잘못된 정보를 고칠 권리
- *삭제 권리* — 정보를 지울 권리 ("잊혀질 권리")
- *이동 권리* — 다른 서비스로 정보를 옮길 권리
- *반대 권리* — 자동 결정에 반대할 권리

AI 엔지니어는 *시스템 설계 시점에 이 권리들을 만족할 수 있는 구조* 를 만들어야 합니다. 예를 들어 *학습 데이터에서 한 사용자만 제거할 수 있는가* 는 매우 어려운 문제입니다 (Machine Unlearning 분야).

### 3.4 챕터 정리

PII 보호는 *익명화·가명화·k-anonymity·DP·Federated Learning·동형 암호* 의 도구로 다룹니다. GDPR·CCPA 같은 규제는 *시스템 설계 시점부터* 권리를 만족할 수 있게 만들어야 합니다.

## 4. 투명성과 설명 가능성

### 4.1 설명 가능성의 종류

- **Global Explanation** — 모델 전체의 동작 설명
- **Local Explanation** — 개별 예측의 근거 설명
- **Feature Importance** — 어떤 입력 특성이 결정에 영향을 미쳤는가
- **Counterfactual** — *X 가 달랐다면 결과가 달랐을 것* 의 분석

규제가 엄격한 분야 (금융 신용·의료 진단·법적 결정) 에서는 *Local Explanation* 이 종종 의무화됩니다.

### 4.2 도구

- **SHAP** — 게임 이론 기반 특성 기여도
- **LIME** — 국소적 선형 근사
- **Integrated Gradients** — 신경망용 그래디언트 기반
- **Attention Visualization** — 트랜스포머의 어텐션 시각화

이 도구들은 *완전한 설명* 을 주지 않습니다. *근사적 설명* 이며, 사용 시 한계를 명시해야 합니다.

### 4.3 챕터 정리

설명 가능성은 *Global·Local·Feature Importance·Counterfactual* 의 종류로 분류되며, SHAP·LIME·IG·Attention Vis 가 표준 도구입니다. *완전한 설명이 아닌 근사* 임을 명시하는 것이 중요합니다.

## 5. AI 안전 (Safety)

### 5.1 LLM 안전 우려

LLM 특화 안전 우려:

- **유해 콘텐츠 생성** — 폭력·성·증오·자해 관련
- **개인 정보 유출** — 학습 데이터 또는 시스템 프롬프트 누출
- **편향된 응답** — 특정 집단에 차별적
- **악용 지원** — 무기·해킹·범죄 정보
- **잘못된 의료·법률 조언** — 자격 없는 전문 조언
- **조작·설득** — 사용자의 의사결정을 부당하게 조작

### 5.2 방어 계층

```
[입력] → 입력 모더레이션 → 모델 → 출력 모더레이션 → [사용자]
                ↑                       ↑
            Use Policy              안전 분류기
```

다층 방어:
- *Use Policy* — 사용 가능 영역 명시
- *입력 필터* — 명백한 악용 의도 차단
- *모델의 RLHF 정렬* — 학습 자체에서 거부
- *출력 분류기* — 유해 응답 자동 차단
- *모니터링과 사후 정지* — 패턴 기반 계정 차단

### 5.3 챕터 정리

LLM 안전은 *유해 콘텐츠·정보 유출·편향·악용 지원·잘못된 조언·조작* 의 6 가지 우려를 다층 방어로 다룹니다.

## 6. 규제 지형

### 6.1 EU AI Act (2024)

세계 최초의 *포괄적 AI 규제 법* 으로, 2024 년 통과. 시스템을 위험도로 4 단계 분류:

1. **금지 (Prohibited)** — 사회 점수 시스템·실시간 생체 식별 (제한적 예외) 등
2. **고위험 (High-risk)** — 고용·신용·의료·법 집행에 영향을 주는 시스템
3. **제한적 위험 (Limited risk)** — 챗봇 (사용자에게 AI 임을 알릴 의무)
4. **저위험 (Minimal risk)** — 대부분의 일반 AI 시스템

고위험 시스템은 *적합성 평가·문서화·모니터링·사고 보고* 가 의무화됩니다.

`[VERIFY: EU AI Act 최종 통과일·발효 일정]`

### 6.2 NIST AI Risk Management Framework

미국 NIST 가 2023 년 발표한 *AI 위험 관리 프레임워크* 입니다. 4 가지 핵심 기능:

- **Govern** — 위험 관리 거버넌스
- **Map** — 시스템과 위험 식별
- **Measure** — 위험 측정
- **Manage** — 위험 대응

법적 의무는 아니지만 *연방 정부 조달 표준* 으로 빠르게 자리잡고 있습니다.

### 6.3 한국·일본·중국

- **한국** — 개인정보보호법 (PIPA) 강화, AI 기본법 논의 중
- **일본** — 자율 규제 중심, 가이드라인 발표
- **중국** — 알고리즘 추천·생성형 AI 별도 규제 (2022, 2023)

각 시장에 진출하려면 *해당 지역의 규제* 를 별도 검토해야 합니다.

### 6.4 챕터 정리

규제는 *EU AI Act (포괄적 법)·NIST AI RMF (자율적 프레임)·국가별 규제* 의 형태로 빠르게 형성 중입니다. AI 엔지니어는 *자기 시스템이 어느 위험도 분류에 속하는지* 를 알고 시작해야 합니다.

## 7. 의사결정 프레임워크

### 7.1 *기술적으로 가능* vs *해야 한다*

새 기능을 만들 때 다음 질문을 던집니다.

1. *기술적으로 가능한가?* — 만들 수 있는가
2. *법적으로 허용되는가?* — 규제·약관 안인가
3. *윤리적으로 정당한가?* — 사용자에게 해를 만들지 않는가
4. *전략적으로 타당한가?* — 회사·사용자에게 가치가 있는가

네 질문 모두에 *예* 일 때만 진행. 한 가지라도 *아니오* 면 재검토.

이 4 단계 점검을 *문서화* 해 두면, 나중에 *왜 이 결정을 했는가* 를 추적할 수 있습니다.

### 7.2 윤리 점검 체크리스트

새 기능 출시 전:

```
□ 사용자에게 만들 수 있는 5 가지 해 분석 완료
□ 편향·공정성 메트릭 측정 완료
□ PII 처리 정책 점검 완료
□ Use Policy 명문화·사용자 고지
□ 법무팀 검토 완료
□ 안전 분류기·모더레이션 활성화
□ 사용자 피드백·신고 채널 마련
□ 인시던트 대응 런북 준비
□ 규제 위험도 분류 (저/제한/고/금지)
```

이 체크리스트를 *PR 머지 게이트* 에 통합하면 자동 적용됩니다.

### 7.3 챕터 정리

*기술적 가능성·법적 허용·윤리적 정당성·전략적 타당성* 의 4 단계 점검과 표준 윤리 체크리스트로 의사결정을 *문서화 가능한 절차* 로 만듭니다.
